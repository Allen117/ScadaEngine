# 功能說明書：能源管理（電表/迴路設定 + 用電報表）

## 1. 功能概述

能源管理模組讓使用者建立**多層樹狀的電表/迴路階層**，並依時/日/月/年四種單位查詢用電量。整體採 **on-demand 即時計算**，不預存加總值，HistoryData 為唯一資料來源。

### 兩個子頁面

| 路由 | 功能 |
|------|------|
| `/EnergyMeter` | 電表/迴路階層 CRUD（樹狀新增、編輯、刪除） |
| `/EnergyReport` | 選一個迴路 + 單位查詢用電，含長條圖 + 表格 + Excel 匯出 |

### 核心概念

- **迴路節點**：階層的最小單元
  - **實體電表**：綁定一個 SID（`ModbusPoints` 或 `CalculatedPoints` 提供累積 kWh），可設 `MaxKwh`
  - **虛擬迴路**：不綁 SID，用電量 = 所有後代葉子的 delta **帶 sign 加權**加總
- **貢獻方向（Sign）**：每個節點對其父節點的方向（+1 / −1，預設 +1）。葉子的有效權重 = 從查詢根到葉路徑上所有 sign 的乘積（不含查詢根本身）。詳見 §5.5
- **單位與期間繫結**（防誤操作 + 控制效能）：

  | 單位 | 期間選擇器 | bucket 數 |
  |------|------------|-----------|
  | 時 | 起訖時（可跨日/月/年） | 起時 ~ 訖時，每小時 1 bucket |
  | 日 | 起訖日（可跨年） | 起日 ~ 訖日，每日 1 bucket |
  | 月 | 起訖月（可跨年） | N 個月 |
  | 年 | 起訖年 | N 年 |

- **kWh 溢位/重置處理**：
  - V_end ≥ V_start：delta = V_end − V_start
  - V_end < V_start 且有 MaxKwh：delta = (MaxKwh − V_start) + V_end
  - V_end < V_start 且無 MaxKwh：delta = 0，記 log warning，回傳結果旗標 `isHasWarning=true`

## 5. 報表算法（關鍵流程）

### Step 1：邊界時刻產生
依單位產出 **N+1 個邊界**（含起點與終點）：

| 單位 | 邊界範例（dtStart=2026-05-05） |
|------|-------------------------------|
| hour | 起時 ~ 訖時隔小時，每小時 1 點（訖時可與起時同小時或跨多時/多日/多年） |
| day | 起日 00:00 ~ 訖日隔日 00:00，每天 00:00（訖日可與起日同日或跨多日/多月/多年） |
| month | 起訖月每月 1 日 + 訖月隔月 |
| year | 起訖年每年 1/1 + 訖年隔年 |

### Step 2：取邊界值（OUTER APPLY 一條 SQL）
對每個葉子，組 `VALUES` 列表加 `OUTER APPLY` 取每個邊界 ≤ 該時刻的最近一筆：

```sql
SELECT b.idx, ba.Value
FROM (VALUES (@i0,@t0), (@i1,@t1), ...) AS b(idx, BoundaryTime)
OUTER APPLY (
    SELECT TOP 1 Value FROM HistoryData WITH (NOLOCK)
    WHERE  SID = @sid AND Timestamp <= b.BoundaryTime AND Quality = 1
    ORDER BY Timestamp DESC
) ba
ORDER BY b.idx;
```

每個 APPLY 命中 `[SID, Timestamp]` clustered seek，O(log N)。

### Step 3：相鄰邊界相減 + 溢位處理 + 加總
- 對每個葉子算出 N 個 delta，套溢位規則（**raw delta 永遠 ≥ 0**，溢位邏輯不受 sign 影響）
- 各葉子的 delta 乘上 effective sign 後同位置相加 → 該迴路該 bucket 的 kWh（**可為負值**）
- 全 bucket 加總 → 總 kWh

### Step 5：貢獻方向（Sign）— 支援 A+B+C−D

每個節點多一欄 `Sign` (+1 / −1，預設 +1)，表示該節點對父節點的貢獻方向。展開葉子時，從查詢根遞迴往下，每層 effective sign = 父 effective sign × 自己 sign（**查詢根本身不計**）。

A+B+C−D 的建模：

```
總計 (Sign=+1, 虛擬, 根)
├── A (Sign=+1, 葉子)
├── B (Sign=+1, 葉子)
├── C (Sign=+1, 葉子)
└── D (Sign=−1, 葉子)
```

報表查詢「總計」時：A、B、C 的 effective sign = +1；D = −1。bucket kWh = A_delta + B_delta + C_delta − D_delta。

**整支子樹反向**：把 D 換成「D 群（含 D1、D2、D3）」，只要 D 群 Sign=−1，其下所有葉子自動反向（−1 × +1 = −1）。

**規則**：

- 根節點（`ParentId IS NULL`）強制 Sign=+1（service / Controller / UI 三層保護），UI 隱藏該選項
- DB 不加 CHECK constraint，由 service / Controller 層驗證 ±1（與專案「不加 FK constraint」慣例一致）
- 溢位規則套在葉子原始 delta（物理意義：實體電表累計值不受 sign 影響），sign 只在最後合併時套用
- 報表 bucket / 子迴路欄位 / Excel 匯出皆會出現負值；Chart.js 不指定 `beginAtZero` 以正確顯示負 bar

**責任歸屬提醒（重要）**：計算點公式內 (`A - B`) 與迴路 sign (Sign=−1) **兩條路都能做減法**。系統不偵測重複扣減；若同一份計算點同時被公式與迴路 sign 扣減，使用者自負其責。建議擇一即可。

### Step 4：Excel 匯出展開直接子（僅匯出路徑）
`/api/export` 走 `GetReportWithChildrenAsync`：先用 Step 1~3 算父，再對父的每個**直接子節點**獨立呼叫一次 `ComputeBucketSumsForCircuit`（內部走相同 Step 2~3）。子若為虛擬節點，`GetLeavesUnderAsync` 仍會遞迴展開葉子，所以子欄就是「該子整棵子樹的合計」。`/api/query`（前端 chart/table）維持原 `GetReportAsync`，不含 children 欄位以節省 payload。實體葉子（自身綁 SID）查詢則略過子層展開，匯出格式維持兩欄。

## 7. UI 操作步驟

### 電表/迴路設定
1. 點右上「+ 新增點表/迴路」建立第一層（例如「水系統」）
2. 在節點上 hover 出現操作鈕，點 + 新增子節點（例如「冰機群 → 冰機 1」）
3. 子節點選擇「實體電表」時，以三層連動下拉挑選點位，再填入 MaxKwh：
   - **通訊設備**：依來源以 optgroup 分組（Modbus 通訊設備 / 計算點位 / DB 來源），僅列出旗下有 kWh 點位的設備
   - **子單元**：僅多 ModbusID 協調器出現此層（列出有 kWh 點位的子設備）；單 ID 協調器、計算點位、DB 來源自動隱藏此層
   - **度數對應點位**：僅列出所選設備（或子單元）下單位為 kWh 的點位
4. **貢獻方向**：新增/編輯時切換「正向 / 反向」決定該節點是被加入還是從父扣減（根節點看不到此選項）
5. **主要電表**：實體電表可勾選「主要電表」，全系統僅能有一顆 — 勾選新的時原本的主要電表自動取消；樹狀清單以金色 ★ 標示、詳情區顯示 badge
6. 反向節點在樹狀清單會顯示紅色 `−` badge 提示
7. 編輯/刪除：用 hover 顯示的 ✎ / 🗑 鈕；刪除有子節點時跳警告
8. 詳情區可看完整資訊（含貢獻方向、主要電表；主要電表另列電壓/電流/功率/功因四個綁定點位）與快速動作

### 主要電表 — 電表資訊設定（電壓/電流/功率/功因）

1. Modal 內勾選「主要電表」後出現「**電表資訊設定**」按鈕（取消勾選即隱藏），按鈕下方顯示已綁定數量摘要（已綁定 n/4）
2. 點按鈕開啟電表資訊 Modal（循序切換，不疊窗）：四列 電壓/電流/功率/功因，各列顯示已綁點位（設備 - 子單元 - 點名 + 單位）或「未設定」
3. **自動比對建議**：開啟時以度數點位名稱為基準，在**相同來源＋通訊設備＋子單元**內比對候選（排除度數點位本身），有候選時頂部顯示建議卡，可「全部帶入」或逐列「帶入」；找不到候選則不顯示建議卡
   - 比對採用與 Designer 列範本共用的 `ROLE_ALIASES` 別名表（`wwwroot/js/role-aliases.js`，掛 `window._roleAliases`），取 V / A / KW / PF 四組
   - 評分規則：點名尾段命中別名 +3、點名整體命中 +3、點名包含別名（≥2 字元）+2、單位命中 +2（電壓 V、電流 A、功率 kW/W/MW、功因 PF/%/空白）、含度數點名 stem +1；分數 ≤ 0 視為無候選，同分取清單序最小者
4. 每列「選擇」開啟**兩步驟選點器**：左側設備清單（多子單元可展開）→ 右側點位清單＋關鍵字搜尋（點名 / SID），已綁定點位自動高亮定位；「清除綁定」可清空該列
5. 四個點位皆可留空（不做必填檢核）；單位一律取綁定點位本身的 Unit，不寫死
6. 4 SID 暫存於前端，隨「儲存」一次寫入 DB；**非主要電表儲存時伺服器端強制清空 4 欄**，資料不殘留
7. **/EMS 顯示**：需量卡片列上方出現 col-12 寬幅「主要電表資訊」卡片 — 主要電表名稱 + 四格（點位名、即時值、單位），即時值走 `POST /api/realtime/by-sids` 每 5 秒輪詢；未綁定欄位顯示「未設定」，無主要電表整卡隱藏（卡片字串已納入 i18n zh-TW/en）

### 用電報表
1. 選一個迴路（樹狀縮排顯示）
2. 選單位，期間選擇器自動切換
3. 按「查詢」→ 看圖表與表格
4. **圖表「顯示明細 / 顯示合計」切換按鈕**（僅虛擬父迴路且直接子 ≥ 2 個時出現於圖表 card-header 右側）
   - 預設「合計」：單色長條，與既有行為一致
   - 切到「明細」：堆疊長條，每個直接子迴路一個 dataset、顏色不同，hover tooltip 列出各子迴路的 kWh + 該 bucket 合計
   - legend 列出所有子迴路名稱對應顏色
   - 子迴路若本身仍是虛擬節點，明細視圖只展開到該子層（其內部仍為合計），與 Excel 匯出語意一致
   - 每次重查強制回到「合計」預設，不保留上次切換狀態
   - 實體電表葉子（自身綁 SID）或無直接子的虛擬節點不顯示此按鈕
5. 按「匯出 Excel」下載 .xlsx（含查詢條件 + 數據表 + 合計）
   - 若選的是**虛擬父迴路**，Excel 額外多出每個**直接子迴路**一欄，逐 bucket 列出該分項 kWh，便於對照各分項佔比
   - 若選的是**實體電表葉子**（自身綁 SID），維持原本「時段 | 用電量」兩欄格式
   - 同列：父迴路欄 ≈ 各子迴路欄總和（受 Math.Round(3) 影響，誤差 < 0.001 kWh）
   - 子迴路若本身仍是虛擬節點，其值為自己整棵子樹的遞迴加總，不再展開

## 9. 設計取捨

- **自參照單表 vs 雙表**：雙表（meter + circuit）需要 join，價值不明，採單表
- **on-demand vs 預聚合**：純 on-demand 在「統計儀表板需查 100 葉子 × 10 年」時會打 ~13,000 次 index seek，冷快取下延遲 5~10 秒 → 加入**葉子層 hourly 預聚合**（見 §11）；階層加總仍 on-demand 保留結構即時生效
- **單位綁固定期間選擇器**：避免使用者選出語意不明的範圍（例如「年單位 + 任意起訖日期」），同時控制 bucket 數量
- **kWh 倒退無 MaxKwh 回 0**：fail-soft，由 log + warning flag 通知使用者補設

- HistoryData 採樣頻率不均時，邊界值可能差到 ±採樣間隔；對 kWh 累積值影響微小（<<0.1%），未特別處理

- 排序與 ParentId 移動的 API 已預留，前端拖曳排序可未來再開

## 11. 葉子層 Hourly 預聚合機制（Engine）

### 11.1 為什麼需要

未來儀表板需要在單頁上同時查「100 個葉子 × 24 小時 × 雙年同期」，純 on-demand 對 HistoryData 跑 OUTER APPLY 在冷快取下會打 ~13,000 次 index seek，延遲 5~10 秒，且每次切時間期間都是冷算。第一性原理：把「葉子 × 小時」這個**最小不可分聚合原子**物化到新表，把查詢成本從 `O(buckets × leaves)` 的 OUTER APPLY 降到一條 `GROUP BY` 的 range scan。**Sign 權重、虛擬節點階層加總仍記憶體做** — 保住「迴路結構改變即時生效不需回填」的紅線。

### 11.2 資料表 `EnergyLeafHourly`

| 欄位 | 型別 | 說明 |
|------|------|------|
| SID | nvarchar(50) | PK，葉子 SID |
| HourStart | datetime | PK，小時起點（local time，與 HistoryData.Timestamp 同基準） |
| DeltaKwh | float | 該小時 kWh 增量（已套溢位、未套 sign） |
| Quality | tinyint | 1=Good, 0=掉線 transition |
| IsRolledOver | bit | 是否觸發 MaxKwh 溢位修正 |
| CreatedAt | datetime | UPSERT 時間 |

### 11.3 Sparse Storage 三段語意

`EnergyLeafAggregator.ComputeAsync(sid, hourStart, maxKwh, maxStalenessHours)`：

- **兩邊界都有值** → 寫入 `Q=1, Delta=計算值`
- **只缺一邊**（staleness 過期或 SID 剛上線首小時）→ 寫入 `Q=0, Delta=0`（給儀表板畫灰 bar 標示掉線 transition）
- **兩邊都缺**（長期斷線中段 / 從沒寫過的 SID）→ **不寫該列**

Web 查詢以 `GROUP BY HourStart` 後在記憶體用 dictionary lookup 補 0（24 個迴圈、零成本），不需 LEFT JOIN。

### 11.4 邊界值 Staleness Window

`appsettings.json:EnergyAggregation.MaxStalenessHours`（預設 2）。`EnergyReportService` 與 `EnergyLeafAggregator` 兩條路徑**都套同一份設定**，確保「當前未完成那小時 on-demand 算」與「上個小時讀 EnergyLeafHourly」的數字連續。

**修正的 bug**：原 `EnergyReportService` 取「最近一筆」無時間上限，電表斷線數天/月後復原會產生「斷線期間每小時 = 0、復原首小時 = 整段累積巨柱」的錯誤。加 staleness window 後：斷線期間 sparse（空白）、恢復首小時 Q=0 灰 bar、恢復第 2 小時起正常 Q=1。

### 11.5 觸發機制

| 觸發 | 何時跑 | 範圍 |
|------|--------|------|
| Hourly Timer | 每小時 XX:02:00 | 上個小時（XX:00 ~ XX+1:00） |
| 啟動 Catch-up | Engine 重啟時 | 過去 `CatchUpHours` 小時（預設 24） |
| MQTT Backfill | 收到 `SCADA/Sys/EnergyLeafHourly/Backfill` | payload 指定 sid + 期間 |

XX:02 而非 XX:00 是給 DB 來源 polling buffer（避免最末幾筆 HistoryData 還沒寫入）。

### 11.6 Backfill MQTT 訊號

```
Topic:   SCADA/Sys/EnergyLeafHourly/Backfill   QoS=1, Retain=false
Payload: { "sid": null | "DB1-S5", "from": "yyyy-MM-dd", "to": "yyyy-MM-dd" }
```

- `sid=null` → 全葉子（典型：首次全量補算）
- `sid="xxx"` → 只算該 SID（典型：MaxKwh 改動後重算歷史）
- 內部按月分批 + 每批 sleep `BackfillBatchSleepMs`（預設 500ms）避免阻塞線上查詢
- UPSERT (MERGE) on `(SID, HourStart)`，與 hourly Timer 重疊也安全

### 11.7 設定區塊（`appsettings.json`）

```json
"EnergyAggregation": {
  "MaxStalenessHours": 2,        // 兩邊都用（Engine + Web）
  "CatchUpHours": 24,            // Engine 啟動補算範圍
  "BackfillBatchSleepMs": 500    // Backfill 每月之間 sleep
}
```

Engine 與 Web 都要設 `MaxStalenessHours` — Engine 用於聚合，Web 用於 on-demand 邊界值查詢。Web 端不需要 `CatchUpHours` / `BackfillBatchSleepMs`。

### 11.8 為什麼 sign / 階層加總不一起預聚合

迴路結構在生產環境會被動 — 新增子表、合併、改變 sign 都很常見。如果連 sign 都預聚合，每次調整都要回填整段歷史；保留「葉子原子」+ 階層 on-demand 計算的設計，**改 sign / 新增虛擬節點 / 改變父子關係即時生效**，不需任何回填動作。葉子層的 MaxKwh 改動是少數需要回填的情境 — 由前述 Backfill MQTT 訊號解決。
