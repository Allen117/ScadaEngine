# 功能說明書 — 演算法服務

> 對應實作：
> - Python：`ScadaEngine.Engine/Algorithms/main.py` + `_status.py` + `*/*.py`
> - C#：`ScadaEngine.Engine/Services/CSharpAlgorithmService.cs` + `Algorithms/_AlgorithmStatus.cs` + `Algorithms/**/*.cs`
> - LogicFlow 整合：`ScadaEngine.Engine/Services/LogicFlowExecutionService.cs`

## 1. 用途

LogicFlow 的 `algorithm` 節點呼叫使用者撰寫的演算法。下游需要分辨：

- `cop=0` 是「真的 0」還是「分母為 0」？
- PID 輸出有沒有觸頂飽和？
- 必填輸入缺漏了嗎？

演算法回傳「結果 + 狀態」結構化 **StatusCode**，下游依嚴重程度決定是否信任、是否切斷下游、是否寫 EventLog。

**設計準則**：演算法寫作者只負責業務邏輯。例外處理（divide by zero、缺輸入、溢位等）由框架（main.py / CSharpAlgorithmService）統一接管，演算法不應寫 try/except / try/catch。

## 2. JSON / API 協定

### Python `POST /algorithms/{name}/evaluate`

請求：
```json
{ "inputs": { "cooling_capacity1": 100, "power1": 25 }, "n": 1 }
```

回應（OK）：
```json
{
  "result": { "cop1": 4.0 },
  "statusCodeId": 0,
  "statusCodeName": "OK",
  "severity": "Info",
  "quality": "Good",
  "perOutput": {
    "cop1": { "statusCodeId": 0, "statusCodeName": "OK", "severity": "Info" }
  }
}
```

回應（變參混合：cop1 除 0、cop2 正常）：
```json
{
  "result": { "cop1": 0, "cop2": 4.0 },
  "statusCodeId": 10,
  "statusCodeName": "DIVIDE_BY_ZERO",
  "severity": "Error",
  "quality": "Bad",
  "perOutput": {
    "cop1": { "statusCodeId": 10, "statusCodeName": "DIVIDE_BY_ZERO", "severity": "Error" },
    "cop2": { "statusCodeId": 0,  "statusCodeName": "OK",             "severity": "Info"  }
  }
}
```

頂層 `statusCodeId/Name/severity` 為「merged」值（取 perOutput 中 severity 最高者，向下相容舊讀取端）。`perOutput` 為 per-output port status map（key 含 variadic suffix），給 Web UI per-port 邊線反灰／Engine per-port HasUpstreamBad 判斷。`quality` 由 merged severity 自動推導（`Error` → `Bad`，其他 → `Good`），保留給舊消費端。

### C# in-process

```csharp
public static AlgorithmResult EvaluateOne(double cooling_capacity, double power)
    => AlgorithmResult.Ok(new() { ["out"] = cooling_capacity / power });
```

## 3. StatusCode 對照表

| codeId | codeName | severity | 觸發情境 | 自動 / 手動 |
|-------:|----------|----------|----------|-------------|
| 0 | OK | Info | 正常 | 自動（預設） |
| 10 | DIVIDE_BY_ZERO | Error | 分母為 0（Python: `ZeroDivisionError`；C#: 任一輸入=0 且結果為 Inf） | **框架自動** |
| 11 | INPUT_MISSING | Error | 必填輸入未提供 / 型別錯（`KeyError` / `TypeError` / `ValueError` / `ArgumentException` / `KeyNotFoundException` / `InvalidCastException`） | **框架自動** |
| 12 | INPUT_OUT_OF_RANGE | Warning | 輸入超出合理範圍但仍可算 | 演算法手動 |
| 20 | SATURATED | Warning | 輸出觸頂 / 觸底（如 PID 飽和） | 演算法手動 |
| 21 | WARMUP | Info | 啟動初期累積樣本中 | 演算法手動 |
| 30 | NUMERIC_OVERFLOW | Error | `OverflowError`／結果為 inf / nan（不是 div by zero） | **框架自動** |
| 40 | DB_ACCESS_FAILED | Error | 演算法需讀寫 DB 失敗 | 演算法手動 |
| 41 | API_CALL_FAILED | Error | 演算法呼叫外部 API 失敗 / Engine 連 Python 服務失敗 | 演算法手動 / LogicFlow |
| 90 | INTERNAL_ERROR | Error | 其他未對應的例外 | **框架自動** |

數字分群（讓未來新增 code 有結構）：

| 區間 | 用途 |
|---|---|
| 0 | OK |
| 1-9 | 保留 |
| 10-19 | 輸入相關 |
| 20-29 | 演算狀態 |
| 30-39 | 數值問題 |
| 40-49 | 外部資源 |
| 50-89 | 保留 |
| 90-99 | 系統錯誤 |

### 框架例外對照表

當演算法拋出下列例外類型時，框架自動轉為對應 status code（演算法**不需自己寫 try/except**）：

| 例外（Python） | 例外（C#） | → status |
|---|---|---|
| `ZeroDivisionError` | `DivideByZeroException` | DIVIDE_BY_ZERO |
| `KeyError` / `TypeError` / `ValueError` | `KeyNotFoundException` / `ArgumentException` / `InvalidCastException` | INPUT_MISSING |
| `OverflowError` | `OverflowException` | NUMERIC_OVERFLOW |
| 其他 `Exception` | 其他 `Exception` | INTERNAL_ERROR |

**結果非有限值**（`IsInfinity` / `IsNaN` / Python `math.isinf` / `math.isnan`）：
- Python：通常 `1.0/0.0` 已先丟 `ZeroDivisionError`；剩餘 inf / nan 來自溢位計算 → NUMERIC_OVERFLOW。
- C#：`double / 0.0` 不丟例外（IEEE 754），由結果推導 — 任一輸入為 0 且結果非有限 → DIVIDE_BY_ZERO；其他 → NUMERIC_OVERFLOW。

## 4. Severity 對下游的影響

| severity | 下游 |
|---|---|
| Info | 結果照常傳遞（含 OK 與 WARMUP） |
| Warning | 結果照常傳遞，但 UI 邊線反灰（橘 `#fd7e14`） + 寫 EventLog |
| Error | 視為 Bad（`HasUpstreamBad`），**該 output port** 的下游節點不再評估、輸出節點本輪跳過不寫 Modbus |

### Per-output 判斷（變參演算法）

`HasUpstreamBad` 對 algorithm 節點走 **per-port** 判斷：依消費端邊線的 `sourcePort` 查 `AlgoPerOutputStatus[sourcePort]`，僅當該 port 的 severity = Error 才視為 Bad。

例：cop_calc N=2，cop1 除 0、cop2 正常 — cop1 邊線下游被擋（output 節點該輪不寫 Modbus），cop2 邊線下游正常傳值 4.0；節點本身只在「**所有** output 都 Error」時整顆反灰。

## 5. EventLog 寫入時機

每個 output port 各自跑「OK ↔ 非 OK」切換偵測：

- `OK → 非 OK`：寫一筆 `EventLog`（EventType=2 Warning 或 EventType=1 Fault，依 severity 對應），`SID = "ALGO:{nodeId}@{treeId}:{outputKey}"`，`Message = "{NodeName}.{outputKey}: {codeName} ({severity})"`。
- `非 OK → OK`：呼叫 `ClearEventAsync(SID)` 標記恢復（同樣帶 `:{outputKey}` 後綴）。
- 連續多次同 code：不重複寫入。

每個 output 各自記錄的好處：變參演算法 N=2 中 cop1 持續 Error、cop2 在 OK ↔ Warning 切換時，兩條時序線分開呈現，不會被 merged 狀態掩蓋。

Severity 映射：`Error` → `EventType=1 Fault, Severity=1 高`；`Warning` → `EventType=2 Warning, Severity=2 中`；`Info` 不寫 log（避免 WARMUP 雜訊）。

訊息直接英文，不走 i18n —— StatusCode 是給開發者/維運看的技術訊息。

## 6. Python 演算法寫法

**唯一支援 `evaluate_one(...)` 介面**。框架負責：從 inputs dict 取對應參數 → 變參迭代 → try/except 對應 status code → 結果 inf/nan 偵測 → 多組 status 聚合（取 severity 最高者）。

### 非變參演算法（如 kWIcalc）

```python
# -*- coding: utf-8 -*-
# @algorithm: kW計算
# @inputs: V, A, PF
# @outputs: out
# @description: 三相功率計算 kW = V * A * PF * 1.732 / 1000

from _status import AlgoStatus, make_status, make_result

def evaluate_one(V, A, PF):
    out = V * A * PF * 1.732 / 1000
    if PF < 0 or PF > 1:
        # 業務語意 status — 演算法主動回傳
        return make_result({"out": out}, make_status(AlgoStatus.INPUT_OUT_OF_RANGE))
    return {"out": out}  # 裸 dict → 視為 OK
```

### 變參演算法（如 cop_calc）

```python
# -*- coding: utf-8 -*-
# @algorithm: COP計算
# @variadic: true
# @inputs_repeat: cooling_capacity:冷凍能力, power:功率
# @outputs_repeat: cop:COP

def evaluate_one(cooling_capacity, power):
    return {"cop": cooling_capacity / power}
```

框架自動：
- 跑 1..n 迴圈，每組從 `inputs["cooling_capacity1"]`、`inputs["power1"]` ... 取值呼叫
- 單組失敗（如 power=0 → ZeroDivisionError）只汙染該組，整體 status 取 severity 最高者
- 結果 dict 自動套 suffix：`{"cop": 4.0}` → `{"cop1": 4.0}`

### 業務語意 status 怎麼回傳

| 情境 | 寫法 |
|---|---|
| 演算法正常 | `return {"out": value}` 或 `return make_result({"out": value})` |
| 業務狀態（SATURATED / WARMUP / INPUT_OUT_OF_RANGE 等） | `return make_result({"out": value}, make_status(AlgoStatus.SATURATED))` |
| 程式技術錯誤（div by zero、缺輸入、溢位等） | **不要自己處理**，讓例外往上拋；框架會接 |

> ⚠️ **舊 `evaluate(inputs)` / `evaluate(inputs, n)` 介面已移除**，不可再用。

## 7. C# 演算法寫法

**唯一支援 `EvaluateOne(double, double, ...)` 介面**。框架透過反射讀參數名 → 從 inputs dict 取對應 key（snake_case key 直接對應 snake_case 參數名）。

```csharp
// @algorithm: COP計算(C#)
// @inputs: cooling_capacity, power
// @outputs: out

using System.Collections.Generic;
using ScadaEngine.Algorithms;

public static class CopCalcCsharp
{
    public static AlgorithmResult EvaluateOne(double cooling_capacity, double power)
        => AlgorithmResult.Ok(new() { ["out"] = cooling_capacity / power });
}
```

業務語意 status：

```csharp
public static AlgorithmResult EvaluateOne(double setpoint, double current)
{
    var output = ComputePID(setpoint, current);
    if (output >= OUT_MAX || output <= OUT_MIN)
        return AlgorithmResult.From(new() { ["out"] = output }, AlgorithmStatusCode.Saturated);
    return AlgorithmResult.Ok(new() { ["out"] = output });
}
```

**重要**：函式參數刻意用 `snake_case`（與 metadata `@inputs` 對齊、與 Python 一致），雖違反 C# PascalCase 慣例但避免雙語言參數命名分歧。

`_AlgorithmStatus.cs` 為共用源碼，編譯時自動隨每個演算法 `.cs` 一起編入（不需手動 import 檔案）。

> ⚠️ **舊 `Evaluate(Dictionary<string,double>)` 介面已移除**，不可再用。

## 8. UI 顯示規則（LogicFlow Web）

- **節點本身**：僅當「所有 output port 都 Error」時整顆套 `.algo-status-bad` + `.algo-status-error` 反灰；混合狀態（部分 Error、部分 OK）下節點維持正常外觀。
- **邊線（per-output）**：依 `_algoStatus.perOutput[edge.sourcePort].severity` 上色：
  - `Info` (OK) → 綠 `#198754` + 值（既有）
  - `Warning` → 橘 `#fd7e14` + `ah-warn` 箭頭 + 值 + hover tooltip
  - `Error` → 紅 `#dc3545` + `ah-bad` 箭頭 + 零填值 + hover tooltip
  - 灰色 `#adb5bd` 既有語意「OK 但訊號未啟動 / 0」維持不動，不在演算法 status 體系內
- **Hover 邊線顯示 tooltip**（SVG `<title>` 子元素）：`{演算法名} : {outputKey} : {codeName} ({severity})`，例 `COP計算 : cop1 : DIVIDE_BY_ZERO (Error)`。
- 所有語系皆顯示英文 codeName，**不走 i18n**。

## 9. 雙語言同步維護

Python 端 `AlgoStatus`（`_status.py`）與 C# 端 `AlgorithmStatusCode`（`_AlgorithmStatus.cs`）為手動同步的兩份對照表，**單一來源為本文件 §3**。新增 code 時：

1. 更新本文件 §3 表格。
2. 同步 `_status.py` 與 `_AlgorithmStatus.cs`。
3. 若新增「框架自動對應」例外類型，同步 `_status.py` 的 `EXCEPTION_STATUS_MAP` 與 `_AlgorithmStatus.cs` 的 `AlgorithmStatus.FromException`。

## 附錄：快速補 metadata header（scada-algo-meta skill）

新增 / 修改演算法常常忘記同步檔頂的 `# @algorithm:` / `// @algorithm:` metadata。專案內附 [scada-algo-meta](../.claude/skills/scada-algo-meta/SKILL.md) skill，由 AST / 正則解析 `evaluate_one` 簽名與 return 抽 inputs/outputs，僅問必要的中文 label + 是否 variadic，**只補缺漏不覆蓋既存**。

**觸發語**：在 Claude Code 對話輸入「幫 cop_calc.py 補 metadata」「為 xxx.cs 加上 header」等。

**支援檔型**：`.py` / `.cs` 兩種；遇到沒有 `evaluate_one` / `EvaluateOne` 的檔案會偵測候選函式並建議 rename。

**改完 metadata 後務必重啟服務**：Python 端重啟 `uvicorn main:app`、C# 端重啟 Engine（`dotnet run`），否則 LogicFlow UI 不會反映新 port/label。

## 10. 已知限制

- 非變參演算法（單一函式回多輸出）：函式拋例外時所有 output 共用同一 Error status（沒辦法只算出其中一組）。
- `STALE_DATA` 由 LogicFlow 上游 Quality 機制處理，不在 StatusCode 範圍內。
- C# variadic 演算法：框架已預留設計（解析 `@variadic`、自動推導 n、迭代呼叫、merge status、per-output map），但目前尚無變參 .cs 演算法用例。
- Edge tooltip 採 SVG `<title>` 子元素，使用瀏覽器原生 tooltip — 出現延遲與外觀不可控；日後若需自訂樣式可改 HTML title attribute + mouseover 自製。
