## 資料流

### Modbus Flow（採集 → MQTT → Web）

```
Modbus TCP Devices
    ↓ FC01/02/03/04 polling (FluentModbus)
ScadaEngine.Engine / ModbusCommunicationService
    ↓ raw value × Ratio → engineering value, quality = Good/Bad
MqttPublishService  (Topic: SCADA/Realtime/{coordinatorName}/{SID}, QoS=1, Retain=true)
    ↓
MQTT Broker (127.0.0.1:1883, no auth)
    ↓ subscribe SCADA/Realtime/+/+
ScadaEngine.Web / MqttRealtimeSubscriberService
    ↓ ConcurrentDictionary<SID, RealtimeDataItemModel>
RealtimeController → /RealTime page
```

Engine 同時寫 SQL Server：`HistoryData`（時序）與 `LatestData`（upsert）。

> **外部系統 / 第三方訂閱 MQTT 整合手冊** → [對外整合_MQTT即時資料訂閱.md](對外整合_MQTT即時資料訂閱.md)（Broker 連線、Topic、Payload JSON schema、Python / Node.js / C# 範例）

### DB 來源 Flow（外部寫入 → Engine polling → SCADA pipeline）

```
外部系統（PLC / 第三方 / 手動匯入）
    ↓ INSERT/UPDATE WHERE SID='DB1-S5' （Value 直接寫工程值）
DBLatestData  (SID, Value, Timestamp, Quality) — 統一入口表
    ↓ polling (per Coordinator PollingInterval, CommandTimeout=ConnectTimeout)：SELECT WHERE SID LIKE 'DB1-%'
DbCommunicationService (Engine BackgroundService)
    ├─ HistoryData：所有已配置 row 餵 HistoryDataStorageService（分鐘 dedup + 60s Timer），Timestamp = polling 當下時間（與 Modbus 一致）
    │   讀失敗（HasEverReadSuccessfully=true 後）→ 對全部 SID 寫 Quality=Bad、Value=LastSeen 最近成功值
    └─ 變化偵測（值/品質/DBLatestData.Timestamp 任一不同才繼續，僅擋下游噪音通道）
        ↓ UPSERT LatestData (Timestamp = DBLatestData.Timestamp)
        ↓ publish MQTT SCADA/Realtime/{Name}/{SID} → 後續警報 / 計算點 / 能源報表 / Web 即時數據
```

SID 命名 `DB{CoordinatorId}-S{Sequence}`（不與 `{ModbusID}-S{N}` 或 `CALC-S{N}` 撞號）；Sequence 由載入器以 JSON 陣列索引+1 自動產生（與 Modbus 對齊，JSON 不顯式指定），每個 Coordinator 上限 100 點。設定唯一來源是 `ScadaEngine.Engine/DBPoint/DB通訊檔案產生工具.xlsm` 的巨集 → `DBPoint/{SheetName}.json`，Engine 啟動讀全部 JSON、UPSERT by Name 到 DBCoordinator/DBPoints；Web 端「DB 來源」頁按「通知 Engine 重新載入 JSON」即發 `SCADA/Sys/DbCoordinator/Reload`（Retain=false），Engine 即時重建 polling loops。**Excel 中加新點位一律加最後一行**，中間插入會位移後續 SID。詳見 [功能說明書_DB來源管理.md](功能說明書_DB來源管理.md)。

> **HistoryData.Timestamp 的取捨**：DB 來源的 HistoryData.Timestamp 用 polling 當下時間（不抄 DBLatestData.Timestamp），與 Modbus 行為一致 — 外部停寫時 HistoryData 仍每分鐘一筆，trending / 用電報表時間軸不斷。LatestData / Realtime / MQTT 的 Timestamp 仍用 DBLatestData.Timestamp 反映外部寫入真實時間。

> **內建 Weather Coordinator（氣象資料來源）**：`DBPoint/Weather.json`（S1=外氣溫度、S2=外氣相對濕度）的「外部系統」是 **Web 端 `WeatherFetchService`** — 依 `/WeatherSetting` 頁設定（CWA 授權碼、測站、間隔）週期呼叫中央氣象署開放資料 API 寫 DBLatestData（Timestamp=觀測時間；API 失敗或觀測 >60 分過舊 → Quality=0）。詳見 [功能說明書_氣象資料來源.md](功能說明書_氣象資料來源.md)。

### OPC UA 來源 Flow（OPC UA Client polling → SCADA pipeline，Web 全欄位動態編輯）

```
外部 OPC UA Server (opc.tcp://..., 帳密驗證, SecurityPolicy None)
    ↓ Session + RegisterNodes；週期批次 Read（依 Server OperationLimits.MaxNodesPerRead 分塊、對齊 Device 邊界）
OpcUaCommunicationService (Engine BackgroundService，每 Server 一條 polling loop)
    ├─ HistoryData：全部點位每輪餵 HistoryDataStorageService（分鐘 dedup）；連線/讀取失敗 → Quality=Bad、Value=最近成功值
    └─ 變化偵測（值/品質變化 或 3 分鐘定期重發，比照 Modbus STALE 防護）
        ↓ UPSERT LatestData → UpdatePersistentCacheOnly → publish MQTT SCADA/Realtime/{Name}/{SID}
        ↓ AlarmMonitorService.EvaluateBatchAsync → CalculatedPointService.CalculateAndPublish
控制寫回：SCADA/Control/{SID} → MqttControlSubscribeService（cid 前綴 OPC 分流）
    ↓ OpcUaCommunicationService.WriteNodeAsync — AO 寫入原始值 = 輸入值 ÷ Ratio、DO 寫 bool；唯讀點位（ControlType 空）拒絕
```

SID 命名 `OPC{CoordinatorId}-S{Seq}`。與 DB 來源最大差異：**設定可在 Web「OPC UA 來源」頁全欄位動態編輯（含新增/刪除 Server、Device、點位），免重啟 Engine**。`OpcUaPoint/{Name}.json` 是 source of truth（一檔一 Server，內含 Devices 分組，點位數不設上限），Web 存檔時：驗證 → 回寫 JSON → UPSERT `OpcUaCoordinator`/`OpcUaPoints`（DB 為執行期快照）→ 發 `SCADA/Sys/OpcUaCoordinator/Reload`（Retain=false）→ Engine `OpcUaReloadSubscriber` 收到後 `ReloadAsync()` 重讀 JSON、重建 Session/polling loops。`Seq` 由 Web 配號並持久化於 JSON（`NextSeq` 單調遞增、刪除不回收）— SID 被 HistoryData/警報/計算點位引用，插入/刪除點位不可位移其他點位 SID。Web 寫檔路徑由 `appsettings.json` 的 `EngineOpcUaConfig`（`WatchedFolder` = Engine 實際讀取的部署資料夾 + 可選 `MirrorFolder` 鏡像寫回 repo）明定，原子寫檔（tmp → `File.Replace` → 留 .bak），與 Modbus 熱編輯線的 `EngineModbusConfig` 同慣例。詳見 [功能說明書_OPCUA通訊.md](功能說明書_OPCUA通訊.md)。

## 資料庫結構初始化與欄位同步

`ScadaEngine.Engine/DatabaseSchema/DatabaseSchema.json` 是**建表 + 欄位的唯一真相來源**。加欄位只改此檔，不再手寫遷移碼。

**執行者**：Engine（`DataServiceExtensions.InitializeDatabaseAsync`）與 Web（`Program.cs` 啟動時）都跑同一個 `DatabaseInitializationService.InitializeDatabaseSchemaAsync` — 消滅部署順序依賴，誰先重啟誰補。Web 端失敗僅 log 不擋啟動（DB 連不上時例外被服務內部攔截、回傳 false）。

**啟動流程**（`DatabaseInitializationService`）：

```
1. 建缺的表        — 逐表查 INFORMATION_SCHEMA.TABLES，缺 → CREATE TABLE（含 PK / 叢集索引）
2. 改名遷移        — MigrateColumnsAsync：僅手寫 sp_rename 段（schema diff 無法推斷改名，看起來像「刪A加B」）
                     必須先於欄位同步，否則新名會被補成空欄位、舊資料孤兒化
3. 欄位同步        — SyncMissingColumnsAsync：逐表比對 INFORMATION_SCHEMA.COLUMNS vs schema
                     缺欄位 → ALTER TABLE ADD（用 schema 的型別/NULL/Default 定義）
```

**同步原則 — 只加不減不改**（生產 DB 零風險；唯一例外是可證明無損的字元欄位加寬）：

| 情境 | 行為 |
|------|------|
| schema 有、DB 缺 | 自動 `ALTER TABLE ADD`，log info |
| DB 有、schema 沒有（人工加過的欄位） | 不動，log warning（漂移可見） |
| 型別不一致 | 不改，log warning（漂移可見） |
| Nullability 不一致 | 不改，log warning（NULL→NOT NULL 需先處理既有 NULL 值，是資料決策） |
| 字元欄位**純加寬**（同型別+同 nullability，如 nvarchar(100)→(500)/MAX） | 自動 `ALTER COLUMN`（無損 metadata 操作），log info；受 PK/索引/約束相依擋下時 log warning 請人工加寬 |
| 字元欄位縮短（DB 比 schema 寬） | 不改，log warning（截斷風險） |
| 缺的欄位是 IDENTITY / PK | 跳過（無法事後 ADD），log error 要求人工遷移 |
| 缺的欄位 NOT NULL 且無 Default | 跳過（對非空表 ALTER 必失敗），log error 要求 schema 補 Default |
| 單欄 ALTER 失敗（Engine/Web 同時啟動 race） | log warning 不中斷整批，另一端已補上即正確 |

機制冪等：欄位齊全時整段無任何 ALTER；未解決的漂移每次啟動持續警告（可見性）。

## 資料庫自動建立與每週備份

核心原則：**安裝時特權與執行時權限分離** — 高權限動作（CREATE DATABASE、建 login、資料夾 ACL）收斂到一次性 `Setting/install-db.ps1`（安裝人員 Windows 帳號 + SSPI，不碰 sa）；日常運行只用 dbSetting.json 的低權限帳號（db_owner）。

`install-db.ps1` 已綁入部署流程自動執行：現場 `Install.bat`（BuildRelease.ps1 產出，步驟 [4/6]、在還原現場設定後）與開發機 `DeployService.ps1` install/update（`Invoke-DbSetup`，QuickDeploy.bat 選項 1/6 都會經過）。失敗不中斷部署，印警告與手動重跑指令。

**自動建 DB 安全網**（`DatabaseInitializationService.EnsureDatabaseExistsAsync`，Engine/Web 共用，掛在建表流程最前）：

```
1. 直接開目標 DB    — 成功即返回（既有環境零負擔）
2. 連 master 查 DB_ID — DB 存在但開不了 → log 權限指引，返回 false
3. DB 不存在 → CREATE DATABASE 於 DbMaintenanceSetting.DataFileFolder + RECOVERY SIMPLE
   ├─ 路徑/ACL 失敗 → 退 SQL Server 預設路徑重試（仍設 SIMPLE），log 警告
   └─ 無 CREATE DATABASE 權限（SqlError 262）→ log「執行 install-db.ps1 或請 DBA 建立」，返回 false
```

既有 DB 完全不動（含復原模式）。新建 DB 一律明確設 `RECOVERY SIMPLE` — 避免繼承 model 的 FULL 導致交易記錄檔無限成長。

**每週自動備份**（`DatabaseBackupService`，Engine BackgroundService）：

- 排程讀 `Setting/DbMaintenanceSetting.json`（預設週日 03:00），主迴圈每 30 秒檢查，同 EnergyLeafAggregationService 模式；失敗當日內每小時重試
- 純 T-SQL `BACKUP DATABASE`，**不依賴 SQL Agent**（Express 無 Agent），db_owner 權限即可
- A/B 兩檔輪替（`{db}_A.bak` / `{db}_B.bak`，`WITH INIT` 覆寫較舊者）— 恆保留本次 + 上一次，磁碟用量可預測
- 壓縮依版本自動判斷：`SERVERPROPERTY('EngineEdition')` 為 2 (Standard) / 3 (Enterprise) 加 `WITH COMPRESSION`，4 (Express) 略過
- 結果寫 EventLog（EventType=3、SID=`_system`；成功 Severity=3、失敗 Severity=2），Web 事件紀錄頁可見
- 啟動自檢：備份資料夾缺失或最新備份逾 8 天 → log 警告，不靜默失敗

**檔案 I/O 歸屬**：CREATE DATABASE 的 MDF/LDF 與 BACKUP 的 .bak 都由 **SQL Server 服務進程**寫入，資料夾需授權 SQL 服務帳號（`NT Service\MSSQLSERVER` 或 `NT Service\MSSQL$實例名`）— 這是 install-db.ps1 的職責之一。DB 在遠端主機時 `TO DISK` 路徑指的是 DB 那台的磁碟。

備份與 DB 檔同機同碟只防「DB 損毀」、防不了磁碟故障 — Backup 資料夾應定期外拷。詳見 [docs/功能說明書_資料庫自動建立與備份.md](功能說明書_資料庫自動建立與備份.md)。

## 警報系統

### Alarm MQTT Topic（未恢復警報即時推播）

```
AlarmMonitorService (Engine) 觸發/恢復
    ↓ AlarmMqttPublisher
SCADA/Alarm/Active/{SID}/{type}   QoS=1, Retain=true
    觸發 → 完整 JSON（severity/message/triggerValue/...）
    恢復 → 空 payload（清除 retained）
    ↓ subscribe SCADA/Alarm/Active/+/+
MqttAlarmSubscriberService (Web) → ConcurrentDictionary<"{SID}:{type}", ActiveAlarmItem>
    ↓ GET /Realtime/ActiveAlarms（記憶體讀取）
前端 active-alarm-panel.js（3 秒輪詢）
```

`{type}` ∈ `high` / `low` / `di`，對應 `EventLog.Operator` 2/3/4。Engine 啟動後 republish 一次目前 active 警報以覆蓋舊 retained；Web 啟動時從 `EventLog WHERE ClearedAt IS NULL AND EventType=0` 預填快取。

### 警報規則異動即時生效（控制 Topic）

```
Web AlarmRuleService (儲存 / 刪除規則) → DB AlarmRules
    ↓ AlarmRuleReloadPublisher
SCADA/Sys/AlarmRules/Reload   QoS=1, Retain=false
    payload: { "sid": "{SID}" | null }
    ↓
AlarmRuleReloadSubscriber (Engine BackgroundService)
    ↓ AlarmMonitorService.ReloadAndReevaluateAsync()
    1) ReloadRulesAsync (含孤立警報清掃)
    2) GetLatestDataAsync 取最新值（含計算點，因為 CalculatedPointService 寫入 LatestData）
    3) EvaluateBatchAsync → 觸發 / 恢復 → 寫 EventLog → 發 SCADA/Alarm/Active/...
```

`Sys` 命名空間預留給未來其他系統級控制訊號（避開 `SCADA/Control/#` Modbus 控制 CID 解析路徑）。Retain=false 避免 Engine 重啟時又跑一次最後一筆 reload；Engine 啟動時 `InitializeAsync` 本來就會做完整 reload，不需 retained 補位。`AlarmMonitorService` 內以 `SemaphoreSlim(1,1)` 序列化 reload 避免快取半更新狀態，並保留原 60 秒 Timer 作為 broker 失聯時的退路。

## 用電報表

### 查詢路徑（On-demand 計算）

```
使用者按「查詢」 → /EnergyReport/api/query
    ↓
EnergyReportService.GetReportAsync(circuitId, granularity, start, end)
    1) GetByIdAsync                取迴路
    2) GetLeavesUnderAsync         遞迴展開所有葉子（綁 SID）
    3) BuildBucketRangesAsync      產出 N 個 bucket 的 [起, 訖) 邊界對
         時/日/年 → BuildBoundaries 連續切界（相鄰 bucket 共用邊界點）
         月       → BillingPeriodService 期別切界（每期各自起訖，可能空窗/重疊，不共用邊界點）
    4) for each leaf:
         OUTER APPLY HistoryData WHERE Timestamp <= boundary
           AND Timestamp >= boundary - MaxStalenessHours AND Quality=1   邊界時刻去重後一條 SQL 取值
         每 bucket 訖值減起值 + 套 kWh 溢位規則 → N 個 delta
       各葉子 delta 同位置相加 → 該 bucket kWh
    ↓
EnergyReportResult { buckets[], totalKwh, isHasWarning }
    ↓
Chart.js 長條圖 + 表格；按「匯出 Excel」 → /EnergyReport/api/export → ClosedXML .xlsx
```

階層存於 `EnergyCircuit`（自參照樹，葉子綁 SID + MaxKwh，虛擬節點不綁）。每個節點有 `Sign` 欄位 (±1)，葉子有效權重 = 從查詢根到葉路徑上所有 sign 的乘積（不含查詢根本身），用以表達「總表 − 已分配子表」「A+B+C−D」這類算式；root 強制 +1，bucket 可為負值。溢位規則仍套在葉子原始 delta 上，sign 只在合併時套用。粒度綁固定期間選擇器（時=日、日=月、月=起訖月、年=起訖年），避免任意起訖造成 bucket 邊界語意混亂。**月粒度的「月」= 月結期別**（YYYY-MM 為期別編號）：期界由 `BillingPeriodService` 依 `BillingPeriods` 表解析（未自訂 = 級聯推導，完全未自訂 = 自然月），用電/冷凍噸/EMS 全站統一；年粒度維持自然年。因月報 on-demand、無月度預聚合表，期別設定改動即時生效、歷史月報自動用新設定重算，不需回填。能源申報為例外 — 固定曆月切界（`GetCalendarMonthlyReportAsync`），不隨期別設定變動。HistoryData 是唯一資料來源，**Engine 只做葉子層 hourly 預聚合**（見下節），**sign / 階層 / 虛擬節點加總仍 on-demand 在 Web 端做**，迴路結構改變即時生效不需回填。詳細功能說明見 [功能說明書_能源管理.md](功能說明書_能源管理.md)。

### 邊界值 Staleness Window

`EnergyReportService.GetBoundaryValuesAsync` 與 `EnergyLeafAggregator` 取「最近一筆 HistoryData」時加上 `Timestamp >= boundary - MaxStalenessHours` 條件（預設 2 小時，`appsettings.json:EnergyAggregation.MaxStalenessHours` 可調）。超過視窗一律視為 null，避免電表斷線復原時把整段累積差異全壓在恢復首小時的「巨柱」假象。**預聚合與 on-demand 必須讀同一份設定**，保住兩條路徑語意一致 — 不然「當前未完成那小時」（on-demand 算）和「上個小時」（讀 EnergyLeafHourly）會出現數字不連續。

### 葉子層 Hourly 預聚合（Engine）

```
Engine BackgroundService (XX:02 觸發 + 啟動 catch-up)
    ├─ EnergyLeafAggregationService → 每葉子 SID 取 [hourStart, hourStart+1hr) 兩邊界值 → 套溢位 → UPSERT
    └─ EnergyLeafBackfillSubscriber (訂閱 SCADA/Sys/EnergyLeafHourly/Backfill, Retain=false)
         payload { "sid": null | "DB1-S5", "from": "yyyy-MM-dd", "to": "yyyy-MM-dd" }
         按月分批 + 每批 sleep BackfillBatchSleepMs (預設 500ms)
    ↓ UPSERT (MERGE on (SID, HourStart))
EnergyLeafHourly (SID, HourStart, DeltaKwh, Quality, IsRolledOver, CreatedAt)
```

**Sparse storage 三段語意**（與 `EnergyLeafAggregator.ComputeAsync` 一致）：
- **兩邊界都有值** → `Q=1, Delta=計算值（含 MaxKwh 溢位修正）`
- **只缺一邊**（staleness 過期或 SID 剛上線）→ `Q=0, Delta=0`（掉線 transition 標記，留給儀表板畫灰 bar）
- **兩邊都缺** → **不寫該列**（避免長期斷線 SID 灌爆空列；Web 端在記憶體用 dictionary lookup 補 0 即可）

觸發時點 XX:02 是給 DB 來源 polling buffer（避免 XX:00 整點時最末幾筆 HistoryData 還沒寫入）。啟動時做 `CatchUpHours`（預設 24）小時的補算覆蓋週末停機等情境。HourStart 用 local time（與 HistoryData.Timestamp 同基準，不轉 UTC）。Engine 完全不做 sign / 虛擬節點加總，那些仍由 Web on-demand 套用，確保迴路結構改變即時生效。Backfill MQTT 訊號典型用途：首次全量補算（sid=null）、單一 SID 在 MaxKwh 改動後重算歷史。

### 電費計算（Web 端逐時計價）

```
EnergyLeafHourly（Engine XX:02 寫入）
    ↓ Web ElectricityCostAggregationService（XX:05 觸發，每輪 DELETE+INSERT 覆蓋近 48h；啟動 catch-up ElectricityCost:CatchUpDays 天，預設 7）
ElectricityCostHourly (SID, HourStart, Period[PK], Kwh, UnitPrice, Cost, PlanId, PlanType, Season, Quality)
    ↓ 查詢：GetLeavesUnderAsync 遞迴 CTE 取 EffectiveSign，per-SID GROUP BY 後記憶體套 sign（同用電報表模式）

POST /ElectricityCostReport/api/query → 電費報表（bucket 切界沿用用電報表；progressive 期別級距分攤）
```

計算放 Web 不放 Engine — TariffConfig 模型 / TariffSettingService / 重算按鈕都在 Web，同一套邏輯直接重用；輸入純 DB 讀取無即時性需求。**只在已選採用方案時計算**（`SystemSettings.electricity_tariff` 的 `szActivePlanId` 非空）。tou 依（DayType × 季節）時段表落段，邊界非整點按分鐘比例分攤拆列（PK 含 Period）；DayType 判定假日優先於星期（`Holidays` 表 → `sun_offday`，由 /HolidaySetting 頁維護）；progressive 只存 kWh、查詢時套級距（避免逐時攤提在重算時的順序相依）。重算（`POST /TariffSetting/api/recalculate`）一律以現行方案費率覆蓋，無費率版本化。詳見 [功能說明書_電費設定.md](功能說明書_電費設定.md) §電費計算、[功能說明書_國定假日設定.md](功能說明書_國定假日設定.md)、[功能說明書_電費報表.md](功能說明書_電費報表.md)。

## 演算法 status 協定

LogicFlow `algorithm` 節點呼叫 Python `/algorithms/{name}/evaluate` 或 in-process C# 演算法時，回傳結構化 StatusCode：

```
result (dict<str,double>)        — 輸出值
statusCodeId / statusCodeName    — 例 10 / "DIVIDE_BY_ZERO"（merged，取 perOutput 中 severity 最高者）
severity                         — Info / Warning / Error
perOutput (dict<port,status>)    — 每個 output port (含 variadic suffix) 的 status
```

`Error` severity 走 **per-output port** 判斷：`LogicFlowExecutionService.HasUpstreamBad` 對 algorithm 節點依消費邊線的 `sourcePort` 查 `AlgoPerOutputStatus[sourcePort]`，**僅該 port 下游**節點不再評估、輸出節點本輪跳過不寫 Modbus；同節點其他 OK port 下游正常傳值。`Warning` / `Info` 仍傳遞結果，僅 UI 邊線反灰 + tooltip。

每個 output port 各自跑「OK ↔ 非 OK」切換偵測寫 EventLog，SID 格式 `ALGO:{nodeId}@{treeId}:{outputKey}`，Severity ≥ Warning 才寫（Info / WARMUP 不寫，避免雜訊）。混合狀態（cop1 Error / cop2 OK）會分兩條時序線記錄，不被 merged 掩蓋。

Python helper：`from _status import AlgoStatus, make_status, make_result`（`_*.py` 不會被當演算法載入）。
C# helper：`using ScadaEngine.Algorithms;`（`_*.cs` 不會被當演算法編譯，而是當共用 SyntaxTree 加入每個演算法的編譯，每個 algo 透過 `AlgorithmResult.Ok(...)` / `AlgorithmResult.From(..., AlgorithmStatusCode.DivideByZero)` 回傳）。

對照表單一來源：[功能說明書_演算法服務.md](功能說明書_演算法服務.md)。

UI：節點本身僅在「**所有 output port 都 Error**」時整顆 `.algo-status-bad` 反灰；混合狀態下節點正常、由邊線各自反灰：Error → 紅 `#dc3545` + `ah-bad`、Warning → 橘 `#fd7e14` + `ah-warn`。Hover 邊線顯示 SVG `<title>` tooltip：`{演算法名} : {outputKey} : {codeName} ({severity})`，**所有語系皆顯示英文 codeName**（不走 i18n）。

## LogicFlow 歷史值讀取

讀取點位（input）與 A/B 接點（contact_no / contact_nc **點位模式**）可勾選「讀取歷史值」，改讀「now − N 分鐘」的 `HistoryData` 值（N = 1–43200，上限 30 天，受 HistoryData 保留期限制）。**預設不勾**：序列化 JSON 不帶新欄位（`histEnabled` / `histOffsetMinutes`），Engine 與前端行為與即時值完全相同（向後相容）。解析度天然為 **1 分鐘**（HistoryDataStorageService 每分鐘存一筆）。

### 資料流（Engine 執行端）

```
HistoryData（每分鐘一筆）
    ↓ 每 1 分鐘：RefreshHistoryCacheAsync()
    │   掃全部 diagrams 收集啟用組合 (SID, offsetMinutes)（Distinct）
    │   每組 GetHistoryTableDataAsync(sid, target−5min, target) 取 ≤ target 最近一筆 Quality=1
    ↓
_historyCache（key=(SID, offset)，snapshot swap，照 _latestCache 模式）
    ↓ 主迴圈（200ms）同步讀快取，不打 DB
四處讀值路徑：GetNodeOutputValue / EvalContact 點位模式 / HasUpstreamBad / HasUpstreamBadFromSource
```

- **查無值（目標時間 ±5 分鐘窗內無 Quality=1）→ 該節點視為 Bad**：走既有 Bad 傳遞語意，下游節點不評估、Output 不寫入。新點或剛上線未滿 offset 時長、HistoryData 已過保留期 → 一律 Bad，屬預期行為。
- DB 每分鐘查詢次數 = 不同 (SID, offset) 組合數；`HistoryData` PK = SID+Timestamp，5 分鐘窗點查走 PK。
- Engine 重啟後歷史讀取立即可用（資料在 DB，首輪 `RefreshCachesAsync` 即刷新，無暖機空窗）。

### 前端（編輯器 + 預覽）

- 點位選擇對話框（picker.js）：checkbox「讀取歷史值」＋「N 分鐘前」數字欄，僅 input / contact 顯示（output、排程模式不顯示）；切排程或非點位節點類型時自動清除欄位。
- 畫布 badge：啟用節點顯示「⏱ -N分」（render.js `histBadgeHtml`，i18n key `logicflow.hist.badge`）。
- 瀏覽器預覽（eval.js）：呼叫 `GET /LogicFlow/api/history-value?sid=&offsetMinutes=`（LogicFlowService 同 Engine 規則查詢），結果依 (SID, offset, 分鐘) 快取於 `S._histCache`，每分鐘最多查一次；查不到 → 節點 `_isBad`、值顯示 `--`。
