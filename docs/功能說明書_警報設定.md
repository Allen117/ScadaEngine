# 功能說明書：警報設定 (AlarmSetting)

## 1. 功能概述

警報設定頁面提供**警報規則的 CRUD 管理介面**，讓使用者為 Modbus 點位或計算點位設定上限警報、下限警報、DI 狀態警報等三種類型的監控條件。設定完成的規則會由 Engine 端的 `AlarmMonitorService` 即時載入並執行監控判定；觸發時除了寫入 `EventLog` 外，也會透過 **Line Messaging API** 推播到設定的群組。

頁面以 **Bootstrap Nav Tabs** 區分為兩個分頁：

| 分頁 | 內容 |
|------|------|
| 警報規則 | 點位警報的閾值、死區、嚴重度等規則 CRUD |
| Line 通知設定 | 收件 Line 群組（GroupID、嚴重度上限）CRUD + 測試發送 |

### 核心能力
- **三種警報類型**：上限（AI ≥ 閾值）、下限（AI ≤ 閾值）、DI 狀態（ON/OFF 觸發）
- **四級嚴重度**：緊急 (0)、高 (1)、中 (2)、低 (3)
- **死區 (Deadband)**：防止臨界值附近頻繁觸發
- **UPSERT 語意**：同一 SID 自動覆蓋，無需手動處理重複
- **即列切換啟停**：表格中直接透過 checkbox 快速切換規則啟停
- **跨模組整合**：ScadaPage 與 Realtime 頁面的即時數據會讀取這些規則進行著色
- **Line 群組推播**：依嚴重度路由到不同群組，每群組獨立限流，Critical bypass，超過則合併摘要

## 4. 資料流架構

```
使用者開啟 /AlarmSetting
    │
    ├─ Controller 查詢 DB ──────────────────────────
    │   ├─ GetAllCoordinatorsAsync()       → 設備清單（Modal 下拉選單）
    │   ├─ GetAllModbusPointsAsync()       → Modbus 點位清單
    │   ├─ GetAllCalculatedPointsAsync()   → 計算點位（合併入點位清單）
    │   └─ AlarmRuleService.GetAllRulesAsync() → 所有規則（JOIN 點位名稱）
    │
    ├─ Razor SSR ──────────────────────────────────
    │   ├─ window._alarmInitData.coordinators  ← JSON 設備陣列
    │   ├─ window._alarmInitData.points        ← JSON 點位陣列
    │   └─ window._alarmInitData.rules         ← JSON 規則陣列
    │
    ├─ JS 初始化 ──────────────────────────────────
    │   ├─ populateCoordinators()  ← 填入設備下拉選單
    │   ├─ renderTable()           ← 渲染規則表格
    │   └─ toggleSection()         ← 初始化各區段顯示/隱藏
    │
    └─ 使用者操作 ─────────────────────────────────
        ├─ 新增規則  → showAddModal() → Modal → save()
        │              → POST /api/alarm-rules → location.reload()
        ├─ 編輯規則  → editRule(id) → Modal → save()
        │              → POST /api/alarm-rules → location.reload()
        ├─ 刪除規則  → deleteRule(id) → confirm()
        │              → DELETE /api/alarm-rules/{id} → location.reload()
        └─ 切換啟停  → toggleEnabled(id, checked)
                       → POST /api/alarm-rules → renderTable()（不重整）

Web 端（規則異動即時通知）：
    AlarmRuleService.SaveRuleAsync / DeleteRuleAsync
        → DB 寫入成功
        → AlarmRuleReloadPublisher.PublishReloadAsync
        → MQTT 發布 SCADA/Sys/AlarmRules/Reload (QoS=1, Retain=false)

Engine 端（獨立 Worker Service）：
    AlarmRuleReloadSubscriber（訂閱 SCADA/Sys/AlarmRules/Reload）
        → AlarmMonitorService.ReloadAndReevaluateAsync
            → 重載 AlarmRules
            → 取 LatestData 全量
            → 對最新值重評（觸發 / 恢復 / 孤立警報清掃）
            → 寫 EventLog + 發 SCADA/Alarm/Active/...

    AlarmMonitorService（保留每 60 秒 Timer 作為退路）
        → 讀取 AlarmRules WHERE IsEnabled = 1
        → 比對即時 MQTT 數據
        → 觸發/恢復警報 → 寫入 EventLog
```

> **規則異動 ~1 秒內生效**：使用者在 Web 改完規則後，Engine 透過 MQTT 立即重載並對 LatestData 重評，前端面板於下一輪 3 秒輪詢即同步消失或更新。詳見 CLAUDE.md「警報規則異動即時生效」段落。

## 5. 控制器說明

### AlarmSettingController

#### `Index()` — 頁面入口

查詢所有設備、點位（含啟用的計算點位）、警報規則，透過 `ViewBag` 序列化為 JSON 注入前端。

#### `GetAllRules()` — GET `/api/alarm-rules`

回傳所有規則（不論啟停），供 ScadaPage 等其他頁面即時著色使用。

#### `SaveRule()` — POST `/api/alarm-rules`

接收 `AlarmRuleSaveDto`，委託 `AlarmRuleService.SaveRuleAsync()` 執行 UPSERT。

#### `DeleteRule()` — DELETE `/api/alarm-rules/{id}`

依 ID 刪除規則，不存在回 404。

## 6. 服務層 (AlarmRuleService)

| 方法 | 說明 |
|------|------|
| `GetEnabledRulesAsync()` | 取得啟用規則（Engine AlarmMonitorService 用） |
| `GetAllRulesAsync()` | 取得全部規則，LEFT JOIN ModbusPoints + CalculatedPoints 帶出 `szPointName` |
| `SaveRuleAsync(dto)` | UPSERT：有 ID → UPDATE by ID；無 ID → 檢查 SID 存在則 UPDATE，否則 INSERT |
| `DeleteRuleAsync(id)` | DELETE by ID |

### UPSERT 邏輯

```sql
-- 有 ID（編輯模式）
UPDATE AlarmRules SET ... WHERE Id = @Id

-- 無 ID（新增模式）
IF EXISTS (SELECT 1 FROM AlarmRules WHERE SID = @SID)
    UPDATE AlarmRules SET ... WHERE SID = @SID
ELSE
    INSERT INTO AlarmRules (...) VALUES (...)
```

此設計確保每個 SID 只有一筆規則。

## 8. 前端邏輯 (alarmsetting.js)

以 IIFE 封裝，對外暴露 `window._alarm`。

### 8.1 初始化

1. 從 `window._alarmInitData` 讀取 Razor 注入的設備、點位、規則資料
2. 初始化 Bootstrap Modal 實例
3. 填入設備下拉選單（含「計算點位」選項，`id = -999`）
4. 渲染規則表格
5. 初始化各區段（上限/下限/DI）顯示狀態

### 8.2 設備/點位聯動選取

三層聯動下拉選單：

```
設備 (selCoord)  →  子設備 (selSubDevice, 選填)  →  點位 (selPoint)
```

**流程**：
1. 選擇設備 → `onCoordChange()`
   - 計算點位 (`-999`)：直接顯示 CALC- 點位
   - 多 ModbusID 設備：顯示子設備欄位，填入子設備下拉選單
   - 單 ModbusID 設備：隱藏子設備欄位，直接篩選點位
2. 選擇子設備 → `onSubDeviceChange()`：依 SubModbusID 範圍篩選
3. 選擇點位 → `onPointChange()`：將 SID 寫入隱藏欄位

**編輯模式**：`setupSelectorsForSid(sid)` 從 SID 反推：
- 計算點位 → 選擇「計算點位」設備
- 一般點位 → 依 SID 數值範圍找到 Coordinator，必要時找子設備

### 8.3 規則表格渲染 (`renderTable`)

| 欄位 | 內容 |
|------|------|
| 啟用 | checkbox，直接觸發 `toggleEnabled()` |
| 設備/點位 | `getFullPointLabel()`：設備名稱 / 點位名稱 |
| 上限警報 | `≥ {閾值}` + 嚴重度 Badge，未啟用顯示 `-` |
| 下限警報 | `≤ {閾值}` + 嚴重度 Badge，未啟用顯示 `-` |
| DI 警報 | `{ON/OFF}` + 嚴重度 Badge，未啟用顯示 `-` |
| 備註 | 純文字 |
| 操作 | 編輯按鈕 + 刪除按鈕 |

- 停用的規則整列灰色 (`table-secondary`)

### 8.4 Modal 表單操作

#### 新增 (`showAddModal`)
- 重置所有欄位為預設值
- 上限/下限/DI 三個區段預設關閉

#### 編輯 (`editRule`)
- 從 `_rules` 陣列找到對應規則
- `setupSelectorsForSid()` 自動定位設備/點位
- 填入所有欄位值
- 根據 checkbox 狀態顯示/隱藏對應區段

#### 儲存 (`save`)
1. 驗證 SID 已選取
2. 組裝 `AlarmRuleSaveDto` JSON
3. `POST /api/alarm-rules`
4. 成功 → 關閉 Modal + `location.reload()`
5. 失敗 → `alert()` 顯示錯誤

#### 區段切換 (`toggleSection`)
- 上限：`chkAlarmHigh` → 控制閾值、死區、嚴重度欄位顯示
- 下限：`chkAlarmLow` → 控制閾值、死區、嚴重度欄位顯示
- DI：`chkDiAlarm` → 控制觸發狀態、嚴重度、ON/OFF 標籤欄位顯示

### 8.5 即列啟停切換 (`toggleEnabled`)

- 不開 Modal，直接送 POST 更新整筆規則（isEnabled 取新值，其餘不變）
- 成功後更新 `_rules` 陣列中的值並重新渲染表格（不重整頁面）

## 9. 三種警報類型

### 9.1 上限警報 (High Alarm)

| 設定 | 說明 |
|------|------|
| 閾值 | 當 AI 點位即時值 ≥ 閾值時觸發 |
| 死區 | 觸發後，需降至 (閾值 - 死區) 以下才恢復 |
| 嚴重度 | 0=緊急、1=高、2=中、3=低 |

### 9.2 下限警報 (Low Alarm)

| 設定 | 說明 |
|------|------|
| 閾值 | 當 AI 點位即時值 ≤ 閾值時觸發 |
| 死區 | 觸發後，需升至 (閾值 + 死區) 以上才恢復 |
| 嚴重度 | 同上 |

### 9.3 DI 警報 (Digital Input Alarm)

| 設定 | 說明 |
|------|------|
| 觸發狀態 | `ON` 或 `OFF`，當 DI 點位進入此狀態時觸發 |
| 嚴重度 | 同上 |
| ON/OFF 標籤 | **唯讀**；自動由 Designer 已發布頁面同 SID 的 DI widget / table cell 標籤帶入。儲存時連同規則保存為快照供 `AlarmMonitorService` 在事件訊息中使用。Designer 內標籤被修改後，下次開啟 AlarmSetting 時會自動同步 |

> **同步機制**：`AlarmSettingController.Index` 載入 `LoadPublishedDesignAsync()` 後解析每頁 `WidgetStateJson`，蒐集 `szType=diPoint` 的 widget props 與 `szType=table` 內 `szPointType=DI` 的 cell，建出 `Dictionary<sid, {onLabel, offLabel}>` 透過 `ViewBag.DiLabelMap` 傳給前端 `_alarmInitData.diLabels`。前端在點位選取/規則編輯/表格渲染時統一以此為準。

## 13. 跨模組整合

### 與 ScadaPage 的整合
ScadaPage 初始化時呼叫 `GET /api/alarm-rules`，將啟用的規則快取到 `_alarmRuleMap[SID]`。Widget 更新時比對閾值，超限則變色（gauge、rtValue、table）或脈動動畫（diPoint）。

### 與 Engine AlarmMonitorService 的整合
Engine 的 `AlarmMonitorService` 每 60 秒呼叫 `GetEnabledRulesAsync()` 重新載入啟用的規則，比對即時 MQTT 數據，觸發時寫入 `EventLog` 事件記錄。

#### 孤立警報自動清除
規則被刪除、整條停用、特定類型旗標關閉（例如 `IsAlarmHigh` 由 1 改 0）或門檻欄位被清空時，原本 `EventLog` 中 `ClearedAt = NULL` 的事件會永遠卡在「警報中」（因為 Engine 不再評估那條規則 → 不會觸發「警報→正常」轉態）。

`AlarmMonitorService` 在以下時機進行**孤立警報清掃**，將失去對應規則的活躍事件以 `ClearedAt = GETDATE()` 自動恢復：
- 啟動時：`InitializeAsync()` 末段（還原 `_alarmStates` 後立即執行一次）
- 運行中：每 60 秒 `ReloadRulesAsync()` 後

清除採用 `ClearEventByOperatorAsync(SID, Operator)`，僅針對特定 `Operator`（2=high / 3=low / 4=di），不會誤清同 SID 但其他類型仍在警報中的事件。

#### Engine 啟動清除系統 Fault 事件
系統型故障事件（`EventType = 1`，例如 LogicFlow 控制寫入連續失敗）依賴記憶體中的計數器追蹤狀態（如 `OutputState.nRetryCount`）。Engine 重啟後計數器歸零，先前未恢復的 Fault 事件無法再以正常流程清除，會永遠掛在 EventLog 中。

`AlarmMonitorService.InitializeAsync()` 開頭會呼叫 `ClearAllUnresolvedFaultEventsAsync()`，以 `UPDATE EventLog SET ClearedAt = GETDATE() WHERE EventType = 1 AND ClearedAt IS NULL` 一次性恢復所有懸掛的系統故障。若實際故障仍存在，相關服務（例如 `LogicFlowExecutionService`）會在累積到觸發門檻後重新寫入新的 Fault 事件，狀態回歸真實。

同時 `GetActiveAlarmsAsync()` 加上 `EventType = 0` 過濾，避免把 Fault 事件還原進使用者規則的 `_alarmStates` 字典造成污染。

### 與 Realtime 頁面的整合
Realtime 頁面目前不直接讀取警報規則，但透過 MQTT 品質欄位（STALE、BAD）間接反映異常。

### 與 ScadaPage 頁面的整合 — 未恢復警報面板 (Active Alarm Summary Panel)
ScadaPage（畫面監控）底部固定顯示一個「未恢復警報」面板，操作員不必切到 EventLog 頁也能掌握當前所有 `ClearedAt IS NULL` 的警報。

資料管線：

```
AlarmMonitorService (Engine)
    ↓ 觸發/恢復時呼叫
AlarmMqttPublisher
    ↓ Topic: SCADA/Alarm/Active/{SID}/{type}, Retain=true
    ↓ 觸發=完整 JSON、恢復=空 payload（清除 retained）
MQTT Broker
    ↓ 訂閱 SCADA/Alarm/Active/+/+
MqttAlarmSubscriberService (Web)
    ↓ ConcurrentDictionary<"{SID}:{type}", ActiveAlarmItem>
    ↓ 啟動時從 EventLog 預填一次（Engine 離線也有資料）
ActiveAlarmController
    ↓ GET /Realtime/ActiveAlarms
前端 active-alarm-panel.js（3 秒輪詢、live counting）
```

排序規則：嚴重度升序 (Critical → Low) → 觸發時間倒序。每筆欄位順序：**觸發時間 → 嚴重度色塊 → 嚴重度標籤 → 訊息 → 距今多久 → 狀態確認 → 確認者**（不顯示 SID/點位名稱，訊息本身已含人類可讀資訊）。Header 提供「全部 / 僅未確認 / 僅已確認」下拉做客戶端篩選；Badge 數仍以「未篩選總數」為準。面板折疊邏輯改採「跨越邊界」原則，避免覆蓋使用者意圖：**首次載入**若無警報則自動折疊；之後僅在警報數 **0 → >0**（新警報出現）時自動展開、**>0 → 0**（全部恢復）時自動折疊；其餘輪詢期間（持續有警報或持續無警報）保留使用者目前手動狀態 — 操作員手動縮起後不會在 3 秒輪詢時被自動展開回去。點選 Header 下拉本身不會觸發折疊。

#### 確認動作 (Acknowledge)
- **操作**：在「未確認」列上**雙擊**即觸發確認動作
- **流程**：JS 呼叫 `POST /Realtime/AcknowledgeAlarm { sid, type }` → Controller 從 `User.Identity.Name` 取出登入帳號 → `MqttAlarmSubscriberService.AcknowledgeAsync` 寫入 `EventLog` (`IsAcknowledged=1`、`AcknowledgedBy=<帳號>`、`AcknowledgedAt=GETDATE()`)，並同步更新記憶體快取
- **DB 篩選條件**：只更新 `SID + Operator + ClearedAt IS NULL + EventType=0 + IsAcknowledged=0` 的列，避免重複確認或誤更新已恢復事件
- **UI 即時回饋**：成功後本地 `lastAlarms` 該筆變為已確認、列轉為灰底；若當前篩選為「僅未確認」則該列即時消失
- **限制**：Web 採單實例假設。如多實例需透過 MQTT 同步 ack 狀態（目前未實作）；下一輪 3 秒輪詢拉到的快取已是更新後值，與 DB 一致

容錯：Engine 啟動完成後會 republish 所有目前 `_alarmStates` 中 active 的警報以覆蓋 broker 殘留 retained；殘留「現已恢復但 retained 還在」的訊息靠下一次該 SID 恢復時清除（接受短暫不一致）。Web 啟動時若 MQTT 尚未連線會先以 EventLog 預填快取。

## 15. Line 通知設定

### 15.1 整體流程

```
警報觸發（AlarmMonitorService.CheckTransitionAsync 狀態翻轉 Inactive→Active）
    │
    ├─ 寫 EventLog（DB）
    │
    └─ LineNotificationService.NotifyAsync(severity, sid, message, dt)
        ├─ 讀 LineNotifyTargets 表（60 秒快取）
        ├─ 篩選符合條件的群組（target.MaxSeverity ≥ alarm.Severity）
        └─ 對每個群組：
            ├─ severity == 0 (Critical) → 直接送、繞過限流
            └─ severity 1/2/3 → 檢查每群組獨立的滑動視窗：
                ├─ Count < 10 → 立即送 + Count++
                └─ Count ≥ 10 → 加入 Buffer
                    └─ 視窗結束（每 5 秒掃描）→ 送摘要訊息（嚴重度計數 + 最近 5 則）
```

### 15.2 設定檔 LineSetting.json

位於 `ScadaEngine.Engine/Setting/LineSetting.json`，**不進 git**（由 .gitignore 排除）。

```json
{
  "ChannelAccessToken": "xxx (Line Channel 長期 token)",
  "EnableNotification": true,
  "RatePerMinute": 10,
  "TestSendThrottleSeconds": 10
}
```

| 欄位 | 預設 | 說明 |
|------|------|------|
| `ChannelAccessToken` | — | Line Messaging API 長期 Channel Access Token |
| `EnableNotification` | `true` | 全域開關。`false` 時所有 Line 推播停用，警報主流程不受影響 |
| `RatePerMinute` | `10` | 每群組每分鐘上限（不含 Critical） |
| `TestSendThrottleSeconds` | `10` | 「測試發送」按鈕同群組的節流秒數 |

**Token 為空或檔案不存在時的行為**：log warning，所有 Line 相關呼叫直接跳過，警報主流程（EventLog 寫入、ScadaPage 著色）完全不受影響。

範例檔 `LineSetting.example.json` 進 git，供新環境部署時參考。

### 15.3 資料表 LineNotifyTargets

```sql
CREATE TABLE LineNotifyTargets (
    Id          INT IDENTITY PRIMARY KEY,
    GroupId     NVARCHAR(64)  NOT NULL,         -- C 開頭 33 字元
    Label       NVARCHAR(100) NOT NULL,         -- 顯示用名稱
    MaxSeverity TINYINT       NOT NULL DEFAULT 3,
    IsEnabled   BIT           NOT NULL DEFAULT 1,
    CreatedAt   DATETIME      NOT NULL DEFAULT GETDATE(),
    UpdatedAt   DATETIME      NULL
);
```

#### MaxSeverity 語意
嚴重度數值越「緊急」越小（0=Critical, 3=Low）。`MaxSeverity` 表示「該群組能接收的最低優先序」：

| MaxSeverity | 接收範圍 | 適用情境 |
|-------------|----------|----------|
| 0 | 只收 Critical | 主管群、值班主管 |
| 1 | Critical + High | 工程組長 |
| 2 | Critical + High + Medium | 工程值班 |
| 3 | 全收 | 全員警報群 |

**篩選規則**：`target.MaxSeverity >= alarm.Severity`，即「群組接收上限的優先序低於或等於該警報嚴重度」。

### 15.4 限流與合併摘要（每群組獨立）

每個 GroupID 各自維護一個滑動視窗計數器：

| 嚴重度 | 行為 |
|--------|------|
| Critical (0) | 永遠單獨送、繞過限流計數 |
| High / Medium / Low (1–3) | 每群組每分鐘 ≤ 10 則 |

#### 摘要訊息格式
當 1 分鐘視窗內超過 10 則時，第 11 則起進入 buffer，視窗結束時統一發送：

```
⚠️ 警報摘要 (近 1 分鐘共 14 則)
視窗: 14:30:00 – 14:31:00
緊急 0 | 高 8 | 中 5 | 低 1

最近 5 則：
• 14:30:42 [高] 溫度超過上限 85.0 (196865-S1)
• 14:30:55 [高] 壓力超過上限 (196865-S2)
...
```

#### 為何「每群組獨立」而非「全域共用」
主管群正常情況只收 Critical（量極少），不該因工程群暴衝（每分鐘超過 10 則）而被拖累拿不到摘要。

### 15.5 重試策略（Engine LineNotificationService）

| HTTP 結果 | 動作 |
|-----------|------|
| 2xx | 成功，結束 |
| 4xx（401/429 等） | **不重試**，log error 後結束（重試只會浪費月額度） |
| 5xx | 等 1 秒後重試 1 次，仍失敗則 log warning 結束 |
| 網路錯誤 / Timeout | 同 5xx，重試 1 次 |

無論失敗與否，**警報主流程（EventLog 寫入）已先完成、不受 Line 影響**。

### 15.6 Engine 啟動順序保證（不重發歷史警報）

```
1. AlarmMonitorService.InitializeAsync()
   ├─ ReloadRulesAsync()
   └─ InitAlarmStatesFromDbAsync() ← 從 EventLog 還原舊警報狀態
                                     此期間 LineNotificationService.IsInitialized = false
                                     即使 NotifyAsync 被誤呼叫也會直接 return
2. LineNotificationService.InitializeAsync(setting)
   ├─ 套用設定
   ├─ 啟動 5 秒 flush 計時器（掃描過期視窗、送 buffer 中的摘要）
   └─ IsInitialized = true ← 此後才開始實際發送
```

### 15.8 測試發送節流

| 項目 | 設計 |
|------|------|
| 節流粒度 | 同一 GroupID（每個群組獨立計時） |
| 節流秒數 | `LineSetting.json` 的 `TestSendThrottleSeconds`（預設 10 秒） |
| 後端實作 | `LineTestSendService` 內 `ConcurrentDictionary<groupId, lastTestAt>`；服務需 Singleton 才能跨請求保留 |
| 觸發時 | 回傳 HTTP **429** + `Retry-After` header + JSON `{retryAfter, message}` |
| 前端反應 | `alert("請稍候 N 秒再試")` |

**為何不在前端做節流**：純前端節流重新整理頁面即可繞過，不安全。

### 15.9 失敗安全（Fail-Safe）原則

下列任一情況皆「不影響警報主流程」：

| 情況 | Engine 行為 |
|------|-------------|
| `LineSetting.json` 不存在 | log warning、`EnableNotification=false`、Notify 直接 skip |
| `ChannelAccessToken` 為空或為佔位符 | log warning、Notify 直接 skip |
| `LineNotifyTargets` 表為空 | Notify 從快取拿到 0 筆、無動作 |
| Line API 回 401（Token 錯誤） | log error、不重試 |
| Line API 回 5xx | 重試 1 次仍失敗則 log warning |
| HttpClient 例外 | log error、Notify 退出（外層 try/catch 阻斷傳遞） |

警報的 `EventLog` 寫入永遠先於 Line 推播，且 `_lineService.NotifyAsync` 包在獨立 try/catch — Line 完全失效時，ScadaPage 著色、EventLog 查詢、Realtime 監控仍正常。
