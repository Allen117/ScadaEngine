# 即時數據（Realtime）功能說明書

> 文件版本：1.0　｜　產出日期：2026-03-31　｜　對應路徑：`ScadaEngine.Web/Features/Realtime/`

---

## 1. 功能概述

即時數據頁面是 SCADA Web 系統的核心監控儀表板，透過 MQTT 訂閱即時接收 Engine 端採集到的所有 Modbus 點位與計算點位數據，並以表格形式呈現。頁面每 **3 秒自動更新**，支援設備篩選、欄位排序、數值變動閃爍動畫。

---

## 2. 頁面路由

| 方法 | 路由 | 說明 |
|------|------|------|
| GET | `/RealTime` | 即時監控主頁面（Server-Side Render） |
| GET | `/api/realtime/latest` | AJAX：取得所有即時資料（前端每 3 秒輪詢） |
| POST | `/api/realtime/by-sids` | AJAX：按 SID 清單查詢（供 LogicFlow 等使用） |
| GET | `/api/realtime/status` | AJAX：連線狀態 + 統計數字 |
| GET | `/api/realtime/point/{subTopic}` | AJAX：單一點位詳細資料 |

## 4. 資料流架構

```
[Modbus / 計算點位 路徑：透過 MQTT]
Engine (Modbus 採集)
  ↓ 發布 MQTT
  ↓ Topic: SCADA/Realtime/{coordinatorName}/{SID}
  ↓ QoS=1, Retain=true
  ↓
MQTT Broker (127.0.0.1:1883)
  ↓ 訂閱 SCADA/Realtime/+/+
  ↓
MqttRealtimeSubscriberService (Web 端 Singleton 背景服務)
  ↓ 解析 JSON → 更新 ConcurrentDictionary 快取
  ↓ ※ SID 以 "DB" 開頭的訊息會被略過

[DB 來源 路徑：直讀 DBLatestData]
RealtimeController.Index / GetLatestData
  ↓ await _mqttService.RefreshDbSourcesAsync()
  ↓ SELECT * FROM DBLatestData WHERE SID LIKE 'DB%'
  ↓ 寫回同一個 ConcurrentDictionary 快取（覆蓋 SID 為 DB 開頭的項目）
  ↓ 品質：SQL 無 exception → GOOD；exception → BAD

[共用後段]
RealtimeController
  ↓ GetAllRealtimeData() 從快取讀取
  ↓
前端 (realtime.js)
  ↓ 每 3 秒 fetch /api/realtime/latest
  ↓ 差異更新表格 + 閃爍動畫
  ↓
使用者瀏覽器
```

## 5. MqttRealtimeSubscriberService（核心服務）

### 5.1 概述

| 項目 | 說明 |
|------|------|
| 類型 | BackgroundService + Singleton |
| 註冊方式 | `AddSingleton` + `AddHostedService`（可同時 DI 注入與背景執行） |
| 訂閱主題 | `SCADA/Realtime/+/+`（即時資料）、`SCADA/LogicFlow/TimerState`（計時器狀態） |
| 快取結構 | `ConcurrentDictionary<SID, RealtimeDataItemModel>` |
| ClientId | `ScadaWeb_Realtime_{ProcessId}`（避免多實例衝突） |

### 5.2 啟動初始化流程

```
ExecuteAsync()
  ├─ 載入 MQTT 配置（MqttSetting.json）
  ├─ 建立 MQTT 連線 + 訂閱主題
  ├─ InitializePointCacheAsync()：
  │   ├─ 從 ModbusPoints 表讀取所有已設定點位
  │   ├─ 從 CalculatedPoints 表讀取所有啟用的計算點位
  │   ├─ 從 LatestData 表讀取每個 SID 的最新值
  │   ├─ 有最新值的點位 → hasData=true，填入實際數值
  │   └─ 無最新值的點位 → hasData=false，Quality="NO_DATA"
  └─ 進入監控迴圈（每 30 秒）：
      ├─ 檢查連線 → 斷線時重連
      └─ CleanupExpiredData() → 超過 5 分鐘未更新標記為 STALE
```

**預填快取的好處**：使用者進入頁面時，即使 Engine 尚未推送新資料，也能看到所有已設定點位的最後已知數值。

### 5.3 MQTT 訊息解析

收到 MQTT 訊息後的解析邏輯：

```
Topic: SCADA/Realtime/Device1/196865-S1
Payload: {"sid":"196865-S1","name":"溫度","value":25.5,"unit":"°C","quality":"Good","timestamp":1711843200000}
```

1. 擷取 subTopic = `Device1/196865-S1`
2. JSON 解析使用**大小寫不分字典**（處理 PascalCase / camelCase 混用）
3. Timestamp 支援兩種格式：Unix 毫秒（long）或 ISO 字串
4. 以 SID 為 key 更新快取（`AddOrUpdate`）

### 5.4 資料過期機制

| 狀態 | 條件 | 處理 |
|------|------|------|
| 正常 | 1 分鐘內有更新 | `isRecent=true` |
| 過時 | 1~5 分鐘未更新 | `isRecent=false`（前端顯示黃色） |
| STALE | 超過 5 分鐘未更新 | `Quality` 標記為 `STALE`（保留最後數值） |
| NO_DATA | 從未收到 MQTT 資料 | `hasData=false`（數值顯示 `--`） |

### 5.5 公開方法

| 方法 | 回傳 | 說明 |
|------|------|------|
| `GetAllRealtimeData()` | `List<RealtimeDataItemModel>` | 取得所有快取資料（依 subTopic 排序） |
| `GetRealtimeDataBySids(sids)` | `List<RealtimeDataItemModel>` | 按 SID 清單篩選 |
| `GetDataStatistics()` | `(total, active)` | 總點位數 + 活躍點位數（1 分鐘內） |
| `IsConnected` | `bool` | MQTT 連線狀態 |
| `PublishControlCommandAsync(cid, value)` | `Task<bool>` | 發布控制指令至 `SCADA/Control/{CID}` |
| `GetTimerStates(treeId)` | `Dictionary` | 取得指定流程圖的計時器狀態 |

## 6. Controller API 說明

### 6.1 GET /RealTime（主頁面）

**回傳**：Razor View + `RealtimeMonitorViewModel`

**處理流程**：
1. 從 MqttRealtimeSubscriberService 取得所有即時資料
2. 從 DB 取得 ModbusCoordinator 清單（左側設備列）
3. 計算統計數字（total/active）
4. Server-Side 先依第一台設備篩選初始資料
5. 將完整資料序列化為 JSON 嵌入頁面（`window._realtimeInitData`）

### 6.2 GET /api/realtime/latest（AJAX 更新）

**回應格式**：

```json
{
  "success": true,
  "timestamp": "2026-03-31 14:30:00",
  "connectionStatus": true,
  "statistics": { "total": 120, "active": 115 },
  "data": [
    {
      "subTopic": "Device1/196865-S1",
      "sid": "196865-S1",
      "name": "冰水出水溫度",
      "value": "7.500",
      "unit": "°C",
      "quality": "GOOD",
      "timestamp": "2026-03-31 14:30:00",
      "isRecent": true,
      "cssClass": "row-recent",
      "badgeClass": "bg-success"
    }
  ]
}
```

**特殊處理**：
- `value`：有資料時顯示小數 3 位（`Math.Round(dValue, 3)`），無資料時顯示 `"--"`
- `timestamp`：有資料時顯示日期時間，無資料時顯示 `"--"`

### 6.3 POST /api/realtime/by-sids（按 SID 查詢）

**用途**：供 LogicFlow 畫布等需要特定點位資料的場景使用，避免回傳全部資料。

**Request Body**：`["196865-S1", "196865-S2", "CALC-S1"]`

**Response**：

```json
{
  "success": true,
  "data": [
    { "sid": "196865-S1", "value": "7.500", "quality": "GOOD" }
  ]
}
```

### 6.4 GET /api/realtime/status（連線狀態）

```json
{
  "success": true,
  "isConnected": true,
  "statistics": { "total": 120, "active": 115 },
  "lastUpdated": "2026-03-31 14:30:00"
}
```

### 6.5 GET /api/realtime/point/{subTopic}（單點查詢）

按 subTopic 查找單一點位的完整資料。找不到回傳 404。

## 8. 前端邏輯（realtime.js）

### 8.1 架構

- **封裝方式**：IIFE `(function(){ ... })()`
- **對外介面**：`window._realtime = { sortTable, refreshData }`
- **自動更新**：頁面載入 1 秒後首次更新，之後每 3 秒輪詢

### 8.2 設備篩選邏輯

左側設備清單點擊後，前端依據 SID 的數字範圍進行篩選（**不需 API 呼叫**）：

```
SID 格式：{nDatabaseId × 65536 + nModbusId × 256 + 1}-S{N}

點擊 Coordinator（全部子設備）：
  範圍 = [CoordinatorId × 65536, CoordinatorId × 65536 + 65536)

點擊子設備（特定 ModbusId）：
  範圍 = [CoordinatorId × 65536 + ModbusId × 256, 同 + 256)

點擊「計算點位」：
  篩選 SID 以 "CALC-" 開頭的項目

點擊 DB 父項目（DBCoordinator 分類）：
  展開/收合子選單，並篩選 SID 以 "DB" 開頭的所有 DB 來源點位（data-sid-prefix="DB"）

點擊 DB 子項目（特定 DBCoordinator）：
  篩選 SID 以 "DB{Id}-" 開頭的項目（data-sid-prefix 驅動）
```

### 8.3 排序功能

點擊表頭可切換排序：

| 欄位 | 排序方式 | 說明 |
|------|---------|------|
| 名稱 | 字串比較 | A-Z / Z-A |
| SID | 字串比較 | — |
| 數值 | 數值比較 | 非數字排到最後 |
| 單位 | 字串比較 | — |
| 品質 | 字串比較 | — |
| 時間戳記 | 字串比較 | 依格式 yyyy-MM-dd HH:mm:ss |

**預設排序**：未點擊任何欄位時，依 SID 中 `-S{N}` 的序號升冪排序。

### 8.4 差異更新機制

為避免每次更新都重建整個表格（造成閃爍），JS 採用智慧判斷：

```
收到 API 回應
  ├─ 將新資料 merge 到 lastData（Map by SID）
  ├─ 篩選當前設備 → 排序
  ├─ 比對新舊 subTopic 順序
  │   ├─ 順序不同 → 全部重建（innerHTML）
  │   └─ 順序相同 → 逐行比對：
  │       ├─ 數值變化 → 更新文字 + 閃爍動畫（cell-updated class）
  │       ├─ 品質變化 → 更新 badge class + 文字
  │       ├─ 時間變化 → 更新文字
  │       └─ CSS class 變化 → 更新列樣式
  └─ 更新頁腳時間 + 連線狀態
```

### 8.5 閃爍動畫

數值變動時觸發 CSS `cell-updated` class：
- 黃色背景閃爍 0.8 秒（`@keyframes cell-flash`）
- 透過 `void el.offsetWidth` 強制重排以重新觸發動畫

## 10. 初始資料載入策略

為了避免頁面載入時出現空白表格，系統採用**雙重填充**策略：

| 階段 | 資料來源 | 說明 |
|------|---------|------|
| 1. 服務啟動時 | DB（ModbusPoints + CalculatedPoints + LatestData + **DBLatestData**） | 預填快取，有歷史值的直接顯示 |
| 2. 頁面載入時 | Server-Side Render（快取內容序列化為 JSON）+ `RefreshDbSourcesAsync` | DB 點位於 Index Action 直讀 DBLatestData |
| 3. MQTT 到達時 | Broker 推送（含 Retain 訊息） | 即時覆蓋快取，**但 SID 以 `DB` 開頭的訊息會被略過** |
| 4. 每 3 秒 | AJAX `/api/realtime/latest` → `RefreshDbSourcesAsync` + 讀快取 | DB 來源直讀 DBLatestData；其他點位由 MQTT 推送維持新鮮 |

**TryAdd 語意**：MQTT 先到的資料不會被 Modbus / 計算點位的 DB 預填覆蓋（先到先贏）。
**DB 來源例外**：DB 點位每次 RefreshDbSourcesAsync 都會 **覆寫**（`_realtimeDataCache[sid] = item`），確保畫面與 DBLatestData 同步。

### 10.1 DB 來源品質判定

`RefreshDbSourcesAsync` 對 `SELECT ... FROM DBLatestData WHERE SID LIKE 'DB%'` 整批 try/catch：

| 情境 | szQuality | dValue | hasData | 顯示 |
|------|-----------|--------|---------|------|
| SQL 成功，DBLatestData 有對應 row | GOOD | row.Value | true | 數值（綠色徽章） |
| SQL 成功，但無對應 row（外部刪除） | GOOD | 0 | true | 0（綠色徽章） |
| SQL exception（連線錯誤等） | BAD | 0 | false | -- |
