# 功能說明書：多語系（zh-TW + en）

## 目的

讓 SCADA Web UI 支援繁體中文 / 英文切換，配合海外或非中文使用者操作。Engine、MQTT、DB 完全 culture-agnostic，語系切換是 Web 進程內部的事。

**未導入 i18n 的頁面**：EnergyMeter / Login / AccessDenied / Profile / ChilledWaterSystem。
- Layout 的側欄/頂欄連結會切換為英文，但點進去的頁面內容仍是中文（已知過渡狀態）

**Designer widget 預設值的 culture 行為**：建立 widget 時依當前 culture 取出預設 `szTitle / szLabel / szText` 等字串並寫入 DB；之後切換 culture 不會改變既有 widget。例如在英文模式新建 Gauge，title 為 "Temperature"，切回中文 UI 仍顯示 "Temperature"。這是 user input snapshot 模型，與「使用者自行命名後系統不再覆寫」的心智模型一致。同理，預設「主頁面 / Main Page」也是建立當下的快照。

**不在 i18n 範圍**：
- DB 內容（點位名、迴路名、使用者輸入文字、警報規則自填的 DiOnLabel/DiOffLabel）
- Serilog 寫到 `ScadaEngine.log` 等檔案的除錯日誌
- **演算法檔的 `@algorithm:` metadata header（displayName）** — 由演算法作者自行決定語言；英文模式下 LogicFlow 節點 displayName 可能仍是中文，此為設計選擇而非 bug（決策見 [`docs/plans/_archive/2026-05-25-conditionctrl-logicflow-i18n.md`](plans/_archive/2026-05-25-conditionctrl-logicflow-i18n.md) 決策 3）

## 架構

### 整體流程

```
使用者點頂欄 fa-globe 下拉 → 選 "English"
   ↓
window.i18n.setCulture('en')
   ↓
POST /api/i18n/set-culture?culture=en      ← 純 Web 內部
   ↓
寫 Cookie ".AspNetCore.Culture=c=en|uic=en"
   ↓
window.location.reload()
   ↓
之後每個 HTTP request 帶 Cookie
   ↓
ASP.NET Core RequestLocalization middleware
   ↓ (Provider 順序：Cookie → QueryString → AcceptLanguage)
CurrentUICulture = en（per request）
   ↓
@Localizer / IStringLocalizer / window.i18n.t() 自動取 .en.resx
```

Engine 沒參與。MQTT broker 沒參與。DB 沒收到 culture 資訊。

### 三層字串來源

1. **Razor server-side 渲染**：`@Localizer["key"]`（`IViewLocalizer`）/ `@SharedLocalizer["key"]`（`IHtmlLocalizer<SharedResource>`），位於 `_ViewImports.cshtml` 注入。
2. **Controller / Service / Excel exporter**：建構子注入 `IStringLocalizer<T>`，存取 `_l["key"].Value`。對於 Singleton 服務（如 `EnergyReportExcelExporter`），若依賴 IStringLocalizer 須改為 Scoped。
3. **JS 端**：`window.i18n.t('key', { args })`，字典由 `GET /api/i18n/{culture}` 端點 fetch 後快取在 `window.i18n._dict`。漏 key 時 fallback 到 zh-TW 字典；都缺則回傳 key 字串並 console.warn。

### resx 檔案組織

- `ScadaEngine.Web/Resources/` 為集中目錄（`AddLocalization` 設定的 `ResourcesPath`）。
- 命名遵循 `{namespace-path}.{culture}.resx` 慣例：
  - 對應 `IViewLocalizer`：`Features.{X}.Views.{ViewName}.resx`
  - 對應 `IStringLocalizer<T>`：`{T-namespace-suffix}.resx`，例 `Features.EventLog.Controllers.EventLogController.resx`
- **中性 .resx 內容 = zh-TW**，搭配 `[assembly: NeutralResourcesLanguage("zh-TW")]`（`Properties/AssemblyInfo.cs`），讓 zh-TW 直接從主組件讀取，不需 zh-TW 衛星組件。en.resx 編譯為 `en/ScadaEngine.Web.resources.dll` 衛星組件。
- Fallback 鏈：`en` → `zh-TW`（中性）。任一 key 漏翻時自動退回 zh-TW。

### key 命名規範

`feature.section.purpose` 全小寫、底線分多字。範例：

- `realtime.col.timestamp`
- `eventlog.event.alarm`
- `shared.button.save`
- `validation.required`
- `alarm.high_exceed`（警報訊息結構化 key — 詳見下節）

## 警報訊息結構化（Engine + DB + Web 跨層）

> 詳細決策見 [`docs/plans/_archive/2026-05-08-i18n-zh-en.md`](plans/_archive/2026-05-08-i18n-zh-en.md) 決策 13、14。

**問題**：Engine 是背景服務，沒有 per-user culture context。同一筆警報訊息可能要顯示給 zh-TW 和 en 兩個使用者。

**解法（Option A）**：Engine 寫入 `EventLog` 時除人類可讀 `Message` 外，另存 `MessageKey`（i18n key）+ `MessageArgs`（JSON 參數），MQTT JSON payload 也帶這兩欄。Web 顯示時透過 `AlarmMessageLocalizer` 依當前 culture 套模板。

### DB schema

`EventLog` 表新增：
- `MessageKey` `nvarchar(64) NULL` — 例 `alarm.high_exceed`；舊資料為 NULL
- `MessageArgs` `nvarchar(512) NULL` — 例 `{"name":"溫度","threshold":"85.0"}`

兩欄定義於 `DatabaseSchema.json`，Engine / Web 啟動時由 `DatabaseInitializationService.SyncMissingColumnsAsync` 自動 ALTER TABLE 補上（schema-driven 欄位同步，詳見 [架構.md](架構.md) §資料庫結構初始化與欄位同步）。

### 三種警報類型固定模板

| nOperator | szType | resx key | 參數順序（{0}/{1}） |
|---|---|---|---|
| 2 | high | `alarm.high_exceed` | name, threshold |
| 3 | low | `alarm.low_below` | name, threshold |
| 4 | di | `alarm.di_triggered` | name, state |

resx 範例（zh-TW / en）：

| key | zh-TW | en |
|---|---|---|
| `alarm.high_exceed` | `{0} 超過上限 {1}` | `{0} exceeds upper limit {1}` |
| `alarm.low_below` | `{0} 低於下限 {1}` | `{0} below lower limit {1}` |
| `alarm.di_triggered` | `{0} 狀態為 {1} 觸發警報` | `{0} state {1} triggered alarm` |

DI 訊息中的 `state` 取自 `rule.szDiOnLabel` / `rule.szDiOffLabel`（使用者自填中文，如「啟動/停機」），fallback 為 "ON"/"OFF"。**屬 user input，不額外翻譯**，直接帶入英文模板。

### 資料流

```
Engine AlarmMonitorService 觸發
   ↓ EvaluateAlarmAsync 組 args = BuildArgsJson(("name", szName), ("threshold", dThreshold))
   ↓ CheckTransitionAsync 寫 EventLog:
      MessageKey  = "alarm.high_exceed"
      MessageArgs = {"name":"溫度","threshold":"85.0"}
      Message     = "溫度 超過上限 85.0"        ← 人類可讀 fallback
   ↓ AlarmMqttPublisher 發 SCADA/Alarm/Active/{SID}/{type}：
      JSON 含 messageKey + messageArgs + message + ...   ← Engine 不知道使用者語系
   ↓
Web MqttAlarmSubscriberService 收到 → ConcurrentDictionary<key, ActiveAlarmItem>
                                        ↑ 含 szMessageKey / szMessageArgs / szMessage
═══════════════════════════════════════════════
   ↓ 使用者開瀏覽器看 Realtime / EventLog
   ↓
HTTP request 帶 Cookie culture=en
   ↓
ActiveAlarmController / EventLogController 取出 (MessageKey, MessageArgs)
   ↓
AlarmMessageLocalizer.Localize(key, argsJson, fallbackMessage)
   ↓ 從 SharedResource resx 撈 "alarm.high_exceed" 對應的 en 模板
   ↓ 解析 argsJson 成位置陣列 [name, threshold]
   ↓ string.Format(template, args) → "溫度 exceeds upper limit 85.0"
   ↓
JSON 回前端
```

舊資料相容：`MessageKey IS NULL` 時 `AlarmMessageLocalizer` 直接回 fallback `szMessage` 原文（中文）。

## 常見維護動作

### 新增一個翻譯 key

1. 找到對應 resx（依 IStringLocalizer<T> 或 IViewLocalizer 的命名規範）
2. **同時** 補 zh-TW（中性 .resx）與 en.resx — 缺一邊會 fallback，但 en 漏翻使用者會看到中文，不該長期殘留
3. JS 端用 `window.i18n.t('key', args)` 引用；server-side 用 `@Localizer["key"]` 或 `_l["key"].Value`
4. 重 build（resx 編進衛星組件）

### 新增一個頁面 i18n

1. 建 `Resources/Features.{X}.Views.{Y}.{resx,en.resx}` 兩份
2. cshtml：把硬寫中文改 `@Localizer["..."]`（不需特別 inject — `_ViewImports.cshtml` 已注入 `IViewLocalizer`）
3. JS：用 `window.i18n.t('...')` 包字串；alert/confirm/innerHTML/template literal 都要包
4. Controller 訊息：建構子注入 `IStringLocalizer<XxxController>`，建 `Resources/Features.{X}.Controllers.{X}Controller.{resx,en.resx}`
5. 在 `docs/i18n-glossary.md` 補新詞（若有）

### 新增第三語言（如日文 ja）

1. `RequestLocalizationOptions.SupportedCultures` 加 `new CultureInfo("ja")`
2. `I18nController._aSupported` 同步加
3. `_Layout.cshtml` 語系切換下拉加新選項
4. 各 resx 補 `*.ja.resx`
5. Engine + DB + MQTT 完全不用動 — 那是 Option A 設計帶來的紅利

### 切換到非英文 culture 時 reload

`window.i18n.setCulture()` 內部用 `window.location.reload()` 整頁重 render — Razor server-side 渲染的字串無法熱替換，reload 是最簡單可靠的做法。Chart.js 等動態元件也會在 reload 後重新建立。

## 注意事項與已知限制

1. **點位名 / 迴路名混雜**：使用者切英文後仍可能看到中文點位名（DB 內容），這是運維責任而非系統責任。
2. **DI 警報訊息中的 state 標籤**：使用者在「警報設定」頁自填的 `DiOnLabel/DiOffLabel`（如「啟動/停機」）視為 user input，直接帶入英文模板 → 英文模式下會出現「Pump1 state 啟動 triggered alarm」這類混雜畫面，非系統 bug。
3. **預留頁過渡狀態**：尚未 i18n 的頁面在英文模式下，側欄連結是英文但點進去內容是中文。
4. **日期時間格式不在地化**：全系統一律 ISO 8601 `yyyy-MM-dd HH:mm:ss`，避免 en-US `5/8/2026` vs zh-TW `2026/5/8` 造成事故。
5. **舊 EventLog 資料**：本次上線前的 EventLog 沒有 `MessageKey` / `MessageArgs`，英文模式下顯示原中文 message — 已知過渡，不做歷史回填。
