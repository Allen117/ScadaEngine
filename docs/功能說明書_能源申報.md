# 功能說明書：能源申報

## 1. 功能概述

EMS（能源管理）體系子頁 `/EnergyDeclaration`。與「用電報表 / 冷凍噸報表」的即席查詢不同，能源申報是**先設定、後產出**：

1. 使用者建立**多份申報報表設定**（如「A 棟申報」「B 棟申報」），每份**成對**綁定：
   - **用電度數來源**：一個 `EnergyCircuit` 迴路節點（kWh，葉子或虛擬皆可）
   - **冷凍噸數來源**：一個 `WaterCircuit` 迴路節點（RT·h，葉子或虛擬皆可）
2. **查詢格式固定**：選定報表 + **申報年度** → 產出該年度 **12 個曆月**（每月 1 號 00:00 ~ 月底）的報表，每月三欄：
   - 用電量 (kWh)
   - 冷凍噸數 (RT·h)
   - 效率 (kWh/RTh) = kWh ÷ RT·h（RT·h ≤ 0 時顯示 `--`）
3. 畫面呈現雙軸圖（kWh 長條 + RT·h 折線）+ 明細表格，可匯出 Excel（ClosedXML）。

### 核心設計原則（第一性原理）

**申報數字必須與單獨報表同一套演算法算出** — 計算層不重寫：

- kWh 委派 `EnergyReportService.GetCalendarMonthlyReportAsync(circuitId, year)`（HistoryData 邊界相減，含 MaxKwh 溢位、Sign、staleness window）
- RT·h 委派 `RefrigerationTonReportService.GetCalendarMonthlyReportAsync(circuitId, year)`（讀 `WaterLeafHourly` 預聚合）
- 兩者皆為既有計算核心（`ComputeBucketSumsForCircuitAsync`）的**曆月切界薄包裝** → 12 個 bucket 依 index 對齊合併（`EnergyDeclarationService` 有防禦性檢查，數量不一致即擲錯）

**曆月 vs 月結期別**：全系統月粒度報表（用電/冷凍噸月報）已改由 `BillingPeriodService` 期別切界；
能源申報是**申報用途**，期界固定為曆月（每月 1 號），刻意**不走**期別 — 這正是兩支 `GetCalendarMonthlyReportAsync` 存在的原因。

## 4. 資料表 `EnergyDeclarationReport`

| 欄位 | 型別 | 說明 |
|---|---|---|
| Id | int identity PK | |
| Name | nvarchar(100) | 報表名稱 |
| EnergyCircuitId | int NOT NULL | 用電來源 — EnergyCircuit.Id（應用層參照，無 DB FK） |
| WaterCircuitId | int NOT NULL | 冷凍噸來源 — WaterCircuit.Id（必填，與用電成對） |
| SortOrder | int default 0 | 新增時自動 MAX+1 |
| Description | nvarchar(500) NULL | |
| CreatedAt / UpdatedAt | datetime | |

由 `DatabaseSchema.json` 啟動自動建表（只加不減）。

**迴路被刪除時**：設定列不做級聯刪除；清單以紅字標示「迴路已不存在」，查詢時回傳明確錯誤訊息請使用者重新編輯設定。

## 5. 合併查詢流程（EnergyDeclarationService.GetDeclarationReportAsync(reportId, year)）

1. 年度範圍檢查（2000~2100）、讀取設定列（不存在 → 擲錯）
2. `EnergyReportService.GetCalendarMonthlyReportAsync(EnergyCircuitId, year)` → 12 個曆月 kWh buckets
3. `RefrigerationTonReportService.GetCalendarMonthlyReportAsync(WaterCircuitId, year)` → 12 個曆月 RT·h buckets
4. 檢查兩邊 bucket 數一致（防未來演算法分岐）
5. 依 index 合併，逐月計算 `dKwhPerRtHour = Round(kWh / RT·h, 3)`；RT·h ≤ 0 → null
6. 總計效率 `dTotalKwhPerRtHour = 總 kWh ÷ 總 RT·h`（總 RT·h ≤ 0 → null）
7. 兩側警告獨立回傳：`isHasKwhWarning`（溢位無 MaxKwh / 缺資料）、`isHasRtWarning`（WaterLeafHourly 覆蓋率 < 90%）

曆月邊界產生器：`EnergyReportService.BuildCalendarMonthRanges(nYear)`（public static，12 對 [每月 1 號, 次月 1 號) + `yyyy-MM` 標籤），兩側服務共用。

## 7. UI 操作步驟

1. EMS 導覽列 → 報表查詢 → 能源申報
2. 上方「申報報表設定」卡片：點「新增報表」→ 彈窗填名稱、選用電迴路、選水系統迴路（兩者必填）、描述（選填）→ 儲存
3. 中間「查詢條件」：選申報報表 + 申報年度（預設今年）→ 查詢，固定產出該年 1~12 月
4. 下方左圖右表：kWh 長條（左軸）+ RT·h 折線（右軸），tooltip footer 顯示該月效率；表格含月份/kWh/RT·h/效率四欄與合計列
5. 「匯出 Excel」產出單檔：報表名稱/兩迴路來源/申報年度/操作者 + 四欄資料表（12 列 + 合計）

## 8. 已知限制

- **查詢格式固定**：只能選年度、固定輸出 12 曆月，無時/日/月/年粒度切換（申報用途，刻意固定）
- **期界固定曆月**：與月結期別（BillingPeriodService）無關；若使用者期望申報月數字 = 用電月報（期別）數字，兩者期界不同會有差異，屬預期行為
- **RT·h 預聚合延遲**：RT·h 來自 Engine 每小時聚合的 `WaterLeafHourly`，最近一小時可能尚未聚合完成，當月（進行中）RT 值可能偏低（沿用冷凍噸報表既有行為，使用者已確認可接受）
- kWh 與 RT·h 資料來源品質不同源（前者 on-demand、後者預聚合），警告旗標各自獨立顯示
- DB 內容（報表名稱、迴路名）為使用者輸入，不在 i18n 範圍
