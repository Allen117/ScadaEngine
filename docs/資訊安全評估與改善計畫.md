## 一、執行摘要

ScadaEngine 的**軟體工程品質良好**（SQL 全面參數化、架構分層清晰、多數輸出正確跳脫），但**資訊安全體質存在系統性缺口**，且因其為可下達實體控制指令的 SCADA 生產系統，風險等級偏高。

本次共發現 **28 項** 具 `file:line` 證據的弱點，分佈如下：

| 嚴重度 | 數量 | 代表項目 |
|:---:|:---:|---|
| 🔴 嚴重 (Critical) | 3 | 控制鏈可完全繞過 Web 認證、控制 API 僅認證未授權、低權帳號可自我提權為 Admin |
| 🟠 高 (High) | 7 | 純 HTTP 明文傳輸、匿名可讀全廠數據、DB 明文弱憑證進版控、無鹽 SHA256、預設帳密、控制值無夾限、儲存型 XSS |
| 🟡 中 (Medium) | 11 | 管理 API 對 User 開放、撤權不即時、無暴防、DB 帳號權限過大、服務跑 LocalSystem、安全標頭全缺、MQTT 無認證、Modbus 網段暴露等 |
| 🟢 低 (Low) | 7 | 授權常數硬編碼、無密碼變更功能、稽核軌跡缺失等 |

**三個必須立即處理的核心問題：**

1. **控制平面無防護（完整性風險最高）**：任何能連上 MQTT broker（1883 明文、匿名）的 LAN 節點，可直接發布 `SCADA/Control/#` 指令，經 Engine 寫入 Modbus 致動泵浦/DO/AO，**完全不需要 Web 登入**。Web 的登入機制不是控制行為的真正信任邊界。
2. **認證 ≠ 授權**：Web 端「只要登入就幾乎能做任何事」。控制 API、帳號管理 API、所有組態/規則/排程 API 都只掛 `[Authorize]`（僅驗證身分、不驗證角色），角色與權限檢查只停留在前端 UI。低權使用者可透過 API 直接致動設備、甚至把自己升級成 Admin。
3. **傳輸與機密全裸**：整站只綁純 HTTP 對外，認證 Cookie 與帳密明文過網；DB 明文弱憑證且已進 git；密碼採無鹽單輪 SHA256。

好消息是：多數問題屬「設定與授權疊加」層次，**修補成本相對可控**，且可分階段止血，不需大改架構。以下依第一性原理拆解並提出可行計畫。

## 二、第一性原理：本系統的「信任邊界」在哪裡？

把系統剝到最底層，資安只需回答一個問題：**「誰、透過哪條路徑、能讓實體設備動作或讀到不該讀的資料？」**

本系統的設計「意圖」是：使用者 → 登入 Web → 依角色/權限操作。但實際的「信任邊界」與意圖不符，出現三處根本性錯位：

- **錯位一：真正的控制信任邊界是「MQTT broker」，而它沒有門。**
  控制決策的最後一哩是「Engine 訂閱 `SCADA/Control/#` → 寫 Modbus」。這一段沒有任何認證。Web 的登入只是「其中一個能發指令的來源」，而非唯一來源。只要 broker 匿名可連，Web 的所有登入/授權機制對「致動設備」這件事等於零。
  → **第一性結論：控制的信任邊界必須下移到 broker（ACL + 認證）與 Engine（指令簽章驗證 + 值夾限），不能只靠 Web。**

- **錯位二：把「認證 (authentication)」當成「授權 (authorization)」。**
  系統有角色（Admin/User）與細緻的頁面/控制權限模型（`PermissionService`），但這些檢查**沒有落到伺服端的寫入/控制 API**——`CanControlScadaPage` / `CanViewScadaPage` 全專案無任何呼叫點，權限只用來在前端隱藏按鈕。伺服端只問「你登入了嗎」，不問「你有權做這件事嗎」。
  → **第一性結論：每一個會改變狀態或致動的伺服端端點，都必須獨立做授權判斷，不能假設「能到達 API = 有權呼叫」。**

- **錯位三：把「網路可達性」當成「安全」。**
  DB / broker 目前多走 loopback，看似安全，但 Web 綁 `0.0.0.0:5038` 對整個 LAN 暴露，且傳輸明文。系統的安全實質建立在「沒人在同網段」這個脆弱假設上，而 OT 網段恰恰是最扁平、最多共用設備的網段。
  → **第一性結論：安全必須來自「加密 + 認證 + 最小暴露面」，而非「大家都在同一個內網、應該沒人會攻擊」。**

本計畫的所有改善項目，都是為了修正這三處錯位。

## 三、系統與部署架構現況

### 3.1 部署拓撲

- 以 `BuildRelease.ps1` 打包 self-contained 版本，現場 `Install.bat` 用 `sc create` 註冊三個 Windows Service：`ScadaEngineService`、`ScadaWebService`、`ScadaEngineLicense`，`start= auto`。
- 三個 `sc create` **均未指定 `obj=` 執行帳號 → 預設以最高權限的 `LocalSystem` 執行**。
- Web (`ScadaEngine.Web/Program.cs:19`) 以 `UseUrls("http://0.0.0.0:5038")` 硬編碼綁定**所有網卡的純 HTTP 埠 5038**；`Install.bat` 另開防火牆放行 5038 inbound → 儀表板直接暴露於整個 LAN。**生產環境無任何 HTTPS endpoint、無反向代理 / TLS 終端。**
- SQL Server、MQTT broker（1883）目前皆設定為本機 loopback（127.0.0.1）。
- Modbus 設備位於 `192.168.147.1:502` 等網段（`Modbus.json`）。

### 3.2 控制資料流（標注每一跳的認證 / 加密狀態）

```
[操作者瀏覽器]
   │  ⚠ HTTP 明文 + Cookie 登入（ControlController 有 [Authorize]，但僅認證未授權）
   ▼   ← 全鏈唯一「有身分驗證」的一跳，且明文可側錄
[ScadaEngine.Web]  PublishControlCommandAsync()
   │  MQTT publish 「SCADA/Control/{CID}」
   │  ❌ 無帳密   ❌ 無 TLS（明文 TCP 1883）   ❌ 無簽章/HMAC   ❌ 無防重放
   ▼
[MQTT Broker 127.0.0.1:1883]
   │  ❌ 匿名可連   ❌ 無 ACL（repo 內無 broker 認證/ACL 設定）
   │  → 任何能連上 broker 的節點都能 publish/subscribe 任意 topic
   ▼
[ScadaEngine.Engine]  MqttControlSubscribeService 訂閱 SCADA/Control/#
   │  ❌ 不驗證來源   ❌ 無白名單/簽章   ❌ 無值範圍夾限（Min/Max 從不套用）
   │  ❌ 整數 unchecked cast（可溢位環繞）
   ▼
[Modbus TCP 設備 192.168.147.1:502]
   │  ❌ Modbus 協定本質無認證、無加密
   ▼
[實體致動：泵浦啟停 / DO 線圈 / AO 暫存器]
```

**關鍵事實：整條控制鏈只有「瀏覽器 → Web」這一跳有身分驗證，且該驗證走明文可被側錄。broker → Engine → Modbus 全程無認證、無加密、無完整性保護。**

### 3.3 認證與授權模型

- Cookie 認證（`ScadaAuth`，4 小時滑動，remember-me 最長 30 天）。登入當下把 `Role`、`Permissions`(JSON) 一次性「烘焙」進 Cookie claims，之後每個請求只讀 Cookie，**不再回查 DB**。
- 全域授權僅掛一個 `PageAccessFilter`，且它**只比對 `ConfigurablePages` 內的「頁面 GET 路由」**（如 `/AccountSetting`），對所有 API 子路由（`/AccountSetting/Create`、`/api/control/write` …）完全放行。
- 全專案 `[Authorize(Roles=...)]` 命中數為 **0**，未設定全域 `FallbackPolicy`。

## 四、威脅模型

| 威脅來源 | 能力假設 | 主要可利用弱點 |
|---|---|---|
| **T1 外部攻擊者（已進入 LAN / VPN）** | 可路由到 5038、1883、502 | 匿名讀全廠數據 (SEC-05)、匿名發控制指令致動設備 (SEC-01)、直連 Modbus (SEC-20)、側錄明文 Cookie (SEC-04) |
| **T2 惡意 / 被盜用的低權內部帳號** | 持有合法 User Cookie | 自我提權為 Admin (SEC-03)、直呼控制 API 致動設備 (SEC-02)、竄改自動控制/警報規則 (SEC-11)、儲存型 XSS 打 Admin (SEC-10) |
| **T3 網段被動竊聽者** | 同網段可嗅探 | 竊取認證 Cookie / 明文帳密 (SEC-04)、竊聽即時與控制訊號 (SEC-19) |
| **T4 原始碼 / 部署包 / DB 外洩** | 取得 repo、Release 包或 DB dump | 明文 DB 憑證 (SEC-06)、離線破解 SHA256 密碼 (SEC-07)、預設帳密後門 (SEC-08)、繞過授權 dongle (SEC-22) |

**最壞情境（多弱點串接）**：T2 低權帳號登入 → 在 Designer 把某頁面命名為 XSS payload (SEC-10) → Admin 開啟 ScadaPage 觸發 → 於 Admin 瀏覽器同源呼叫 `/api/control/write` (SEC-02) → 致動關鍵設備；或 T1 直接對 broker 注入控制指令 (SEC-01)，全程不需任何帳號。

## 五、弱點清單（依嚴重度）

> 格式：**[編號] 標題** — 位置 / 說明與攻擊情境 / 建議修復。

### 🔴 嚴重 (Critical)

**[SEC-01] 控制鏈可完全繞過 Web 認證（MQTT broker 匿名明文 + Engine 訂閱端零授權）**
- 位置：`ScadaEngine.Engine/Communication/Mqtt/MqttControlSubscribeService.cs:148`（`.WithTcpServer` 無 `.WithCredentials`/`.WithTls`）、`:222`（訂閱 `SCADA/Control/#`）、`:288` 起（收訊即解析→寫 Modbus，無來源驗證）；`ScadaEngine.Engine/Models/MqttConfigModel.cs`（無 username/password/TLS 欄位）。全解決方案 grep `WithCredentials`/`WithTls` 為零命中。
- 說明／情境：broker 匿名、明文 1883、無 ACL；Engine 訂閱端不驗證來源、不驗簽章。**任何能連上 broker 的 LAN 節點，用任一 MQTT 工具發布 `SCADA/Control/{CID}` 即可致動實體設備**，完全繞過 Web 登入。可掃描 CID 對所有可寫點位下指令。
- 建議：broker 啟用帳密 + TLS(8883) + per-client ACL（僅 Web client 可 publish 控制前綴，Engine 僅可 subscribe）；Engine 端驗證指令簽章。（見改善計畫 P0-1 / P1-3）

**[SEC-02] 控制 API `/api/control/write` 僅認證未授權 — 任何登入者可致動設備**
- 位置：`ScadaEngine.Web/Features/ScadaPage/Controllers/ControlController.cs:12`（僅 `[Authorize]`）、`:36 Write`；`PermissionService.cs:119/131`（`CanViewScadaPage`/`CanControlScadaPage` 定義）。全專案 grep 證實此二方法**無任何伺服端呼叫點**，控制權限只在前端 JS 隱藏右鍵選單。
- 說明／情境：低權 User（甚至僅檢視權）用 DevTools / curl 帶自己合法 Cookie 直接 `POST /api/control/write {"cid":"...","value":...}`，即可啟停泵浦、改 AO/DO、切手自動，繞過 UI 隱藏。
- 建議：`ControlController.Write` 內以 CID 反查所屬 ScadaPage SID，伺服端強制呼叫 `CanControlScadaPage`，非授權回 403。（P1-2）

**[SEC-03] 帳號管理 API 無角色限制 → 低權帳號可自我提權為 Admin**
- 位置：`ScadaEngine.Web/Features/AccountSetting/Controllers/AccountSettingController.cs:8`（類別僅 `[Authorize]`）、`:29 Create`、`:44 Update`、`:64 Delete`、`:77 GetPermissions`；`ScadaEngine.Web/Services/AccountSettingService.cs:54-84`（`UpdateUserAsync` 僅驗證角色字串為 Admin/User 即放行）、`:89-123 RefreshAuthCookieIfSelfAsync`（改到自己時當場換發帶新角色的 Cookie）。
- 說明／情境：這些 POST 不在 `ConfigurablePages`，`PageAccessFilter` 不攔；Controller 又無角色限制。**User 直接 `POST /AccountSetting/Update` 帶自己的 `UserID` + `Role="Admin"`，回應即刻換發 Admin Cookie**，瞬間變管理員；亦可 `Create` 後門 Admin、`Delete` 掉所有 Admin、`GetPermissions/{任意ID}` 讀他人權限（IDOR）。
- 建議：`AccountSettingController` 全類別加 `[Authorize(Roles="Admin")]`；`UpdateUserAsync` 禁止提升自身角色；伺服端驗證 `UserID` 歸屬。（P0-3）

### 🟠 高 (High)

**[SEC-04] 純 HTTP 明文傳輸，HTTPS / HSTS / Cookie Secure 全數失效**
- 位置：`Program.cs:19`（`http://0.0.0.0:5038`）、`:83`（`Cookie.SecurePolicy = SameAsRequest`）、`:212 UseHsts()`、`:216 UseHttpsRedirection()`；`appsettings.json`（`AllowedHosts:"*"`）；`Install.bat` 防火牆放行 5038。
- 說明／情境：無 HTTPS 監聽埠 → `UseHttpsRedirection` 為 no-op、HSTS 標頭在 HTTP 回應被瀏覽器忽略、`SameAsRequest` 使 Cookie 不帶 `Secure` → 認證 Cookie、登入帳密、控制指令全程明文。同網段被動嗅探即可劫持 Session（含 Admin）或擷取明文帳密。程式表面「看起來有 HTTPS」實則全失效，形成危險假象。
- 建議：反向代理（IIS/Nginx）終結 TLS，或 Kestrel 綁 HTTPS + 正式憑證；`SecurePolicy=Always`、顯式 `SameSite`；限縮 `AllowedHosts` 與監聽介面。（P1-1）

**[SEC-05] 匿名可存取即時數據 / 警報清單 / 授權驗證 API（無全域授權後備）**
- 位置：`RealtimeController.cs:11-12`（類別無 `[Authorize]`，含 `/api/realtime/latest|by-sids|status|point`）；`ActiveAlarmController.cs:10-11`（GET `ActiveAlarms` 匿名，僅 `AcknowledgeAlarm` 有 `[Authorize]`）；`LicenseApiController.cs:14-16`（`GET status` / `POST verify` 匿名）；`Program.cs` 未設 `FallbackPolicy`。
- 說明／情境：未登入的外部攻擊者（此為線上生產機）直接 `GET http://host:5038/api/realtime/latest` 即可持續擷取全廠即時 SID/名稱/數值/單位/品質做工控偵察；讀警報清單得知異常設備與門檻；匿名反覆 `POST /api/license/verify` 每次觸發新 broker 連線形成資源耗用。
- 建議：三個 Controller 加 `[Authorize]`；`Program.cs` 加全域 `FallbackPolicy = RequireAuthenticatedUser()`（一次堵住所有「忘了加 [Authorize]」的匿名破口）。（P0-4）

**[SEC-06] DB 明文弱憑證且已納入 git 版控**
- 位置：`ScadaEngine.Engine/Setting/dbSetting.json`、`ScadaEngine.Web/SQLSetting/dbSetting.json`（兩份皆 git 追蹤、有歷史提交）；`.gitignore` 的「Sensitive config」排除規則被**註解掉**（對照 `LineSetting.json` 有正確排除）；`BuildRelease.ps1` 另將此檔複製進 `Release/` 部署包。
- 說明／情境：DB 帳密以明文、且為短低熵弱憑證存放，並永久留存於 git 歷史與離線部署包（USB 交付即實體外洩）。無任何 DPAPI/加密/Key Vault。
- 建議：改整合式驗證或受管身分；否則以 DPAPI/環境變數/密鑰保管庫存放，設定檔移出版控，**輪換現有密碼（視為已洩漏）**。（P0-5 / P2-8）

**[SEC-07] 使用者密碼採無鹽、單輪 SHA256**
- 位置：`ScadaEngine.Engine/Data/Repositories/SqlServerDataRepository.cs:619`（`ValidateUserAsync`）、`:687`（`CreateUserAsync`）——全系統僅有的兩個雜湊點。
- 說明／情境：`SHA256.HashData(...).ToLower()`，無 salt、無 pepper、單輪、快速雜湊。`Users.PasswordHash` 一旦外洩，可用彩虹表 / GPU 高速離線破解；相同密碼產生相同 hash，可辨識共用密碼帳號。
- 建議：改用 ASP.NET Core `PasswordHasher<T>`(PBKDF2) 或 bcrypt/Argon2，per-user salt + 高迭代；登入成功時就地透明升級舊 hash 平滑遷移。（P1-5）

**[SEC-08] 硬編碼預設管理者帳密 admin/admin**
- 位置：`LoginController.cs:20-21`（常數）、`:67-71`（`GetAdminCountAsync()==0` 時生效）；`SqlServerDataRepository.cs:580`（Admin 計數＝`Role='Admin' AND IsActive=1`）。
- 說明／情境：當 DB「啟用中的 Admin 數＝0」時，眾所周知的 `admin/admin` 可登入並取得 Admin 全權。全新部署、DB 誤刪、或所有 Admin 被停用期間即為公開後門；帳密為程式常數無法從 UI 變更，且源碼一旦外流等同後門。（另：部署文件標示的預設帳密與程式碼常數不一致，運維易誤判。）
- 建議：移除硬編碼 fallback，首次啟動強制建立 Admin 並設定強密碼；若保留 bootstrap 須首登強制改密並自動失效。（P0-2）

**[SEC-09] Engine 控制值無範圍夾限，且整數轉換為 unchecked cast 會溢位**
- 位置：`MqttControlSubscribeService.cs`（`WriteIntegerRegisterAsync` `(short)(dValue/fRatio)`、`WriteUIntegerRegisterAsync` `(ushort)` 截斷、Coil `dValue>0.5`）。點位模型 `ModbusModels.cs` 有 `szMin/szMax` 且 `Modbus.json` 每點都設了值，但**寫入路徑從頭到尾未讀取或比對 Min/Max**（grep `fMin/fMax/Clamp/checked` 於 Mqtt 資料夾零命中）。同模式亦見於 `LogicFlowExecutionService.cs`、`ConditionControlService.cs`。
- 說明／情境：對 AO 寫入遠超量程的值（如頻率 9999），或利用 `(short)` 溢位讓 70000 環繞成負值，設備收到非預期指令造成超速/跳機/機械損害。即使合法 Web 操作，缺後端夾限也可能因前端疏漏放行危險值。
- 建議：寫入前一律以點位 `fMin/fMax` clamp 並拒絕+記錄超範圍指令；整數轉換改 `checked{}` 或先驗證範圍。抽共用安全寫入函式一併修補三處。（P1-4）

**[SEC-10] 儲存型 XSS — 畫面樹狀節點名稱／圖示未跳脫即寫入 innerHTML**
- 位置：`ScadaEngine.Web/wwwroot/js/scadapage.js:200-202`（`page.szName`/`page.szIcon` 未經 `escViewHtml()` 直接拼接後 `el.innerHTML = szHtml`）。同檔其他 widget 建構皆有正確呼叫 `escViewHtml`，唯獨樹節點漏掉。
- 說明／情境：`szName`/`szIcon` 來自 Designer（一般 `[Authorize]` 即可存取）的使用者輸入並存 DB。把某頁面命名為 `<img src=x onerror=...>`，之後**任何開啟 ScadaPage 者（含 Admin）瀏覽器都會執行** payload；因屬同源腳本，可繞過 CSRF 的 SameSite 緩解，直接驅動 `/api/control/write` 操作設備。
- 建議：`escViewHtml(page.szName)`；`page.szIcon` 以白名單校驗（僅允許 `^[a-z0-9- ]+$` 的 FontAwesome class）。全面 review 其餘 innerHTML 拼接點凡 DB 來源字串皆跳脫。（P1-7）

### 🟡 中 (Medium)

**[SEC-11] 所有「設定 / 規則 / 迴路 / 排程」管理 API 可被 User 角色呼叫**
- 位置（代表）：`AlarmSettingController.cs`（Line/Email/SMTP/警報規則 CRUD，含 `GET email-config` 回傳 SMTP 設定）、`ScheduleSettingController.cs:46/118`、`CalcPointController.cs:75/87/99`、`LogicFlowController.cs:49-95`、`ConditionCtrlController.cs:60`、`ModbusCoordinatorController.cs:30`、`DbCoordinatorController.cs:58`、`DesignerController.cs:107/138`、`EnergyMeterController.cs:125-162`、`ChilledWaterSystemController.cs:121-149`、`EventLogController.cs:107 trigger / :140 clear`。全部僅類別 `[Authorize]`。
- 說明／情境：低權 User 可修改條件控制/流程圖規則改變自動控制行為、刪除警報規則使關鍵告警失效、改 SMTP 設定把通知導向攻擊者、灌假警報或 `clear` 掉真實警報做告警抑制掩護實體操作。
- 建議：依「管理即 Admin」原則，為上述管理型 Controller 統一加 `[Authorize(Roles="Admin")]`（如需細分則導入以 `ConfigurablePages` 為基礎的伺服端 policy）。（P1-2）

**[SEC-12] 停權 / 降級 / 改權 / 刪除對既有 Session 不生效**
- 位置：`LoginController.cs:102-118`（Role/Permissions 烘焙進 claims、remember-me `AddDays(30)`）；`Program.cs:79-84`（無 `OnValidatePrincipal`）；`IsActive` 僅登入時查（`SqlServerDataRepository.cs:608-610`）；`RefreshAuthCookieIfSelfAsync` 只刷新「操作者自己」。
- 說明／情境：角色與權限只在登入瞬間寫入 Cookie，其後不回查 DB 也無 security stamp。把帳號停權、降級、刪除後，該使用者的**現有 Session 仍持舊權限有效**，remember-me 情境最長 30 天。離職或帳號遭盜用無法即時關閉。
- 建議：設定 `CookieAuthenticationEvents.OnValidatePrincipal`，每數分鐘回查 DB 的 `IsActive`/`Role`/權限版本（security stamp），不符即 `RejectPrincipal` + `SignOut`。（P2-1）

**[SEC-13] 登入無暴力破解防護**
- 位置：`LoginController.cs:48-79`（失敗僅 `LogWarning`，無鎖定/次數上限/速率限制/CAPTCHA/延遲）；監聽對外 `0.0.0.0:5038`。
- 說明／情境：可對任意帳號無限次線上字典/暴力攻擊，配合 SEC-07 快速雜湊更省算力。
- 建議：帳號/IP 失敗計數 + 鎖定與指數退避、登入速率限制、連續失敗後 CAPTCHA。（P2-2）

**[SEC-14] DB 應用帳號權限過大（具 DDL 權）＋ SQL 帳密 ＋ TrustServerCertificate=true**
- 位置：`DatabaseInitializationService.cs`（啟動時自動 `CREATE TABLE`，推測帳號至少 `db_ddladmin`/`db_owner`）；連線建構 `ScadaEngine.Common/Data/Models/DatabaseConfigModel.cs:40`（`Microsoft.Data.SqlClient` + 硬編碼 `TrustServerCertificate=true`、未顯式 `Encrypt`）。
- 說明／情境：runtime 僅需 CRUD 卻握有 DDL 級高權帳號，一旦應用被攻陷即取得高權 DB 帳號；`TrustServerCertificate=true` 停用伺服器憑證驗證，DB 一旦遷至遠端即暴露 TLS 中間人風險。
- 建議：分離「初始化（高權、一次性）」與「執行（最小 CRUD）」兩組帳號；`Encrypt=True` 並移除 `TrustServerCertificate`（部署受信任憑證），至少改為可由設定控制。（P2-3）

**[SEC-15] 三個 Windows Service 以 LocalSystem 執行**
- 位置：`Install.bat`（`sc create` 未帶 `obj=`，Engine/Web/License 三服務皆 LocalSystem）。
- 說明／情境：違反最小權限；面向 LAN 的 Web 服務以 SYSTEM 執行，遭 RCE 時直接取得本機最高權限。
- 建議：改用專屬低權服務帳號（gMSA 或 `NT SERVICE\...`），僅授應用資料夾與必要資源存取權。（P2-4）

**[SEC-16] 安全性回應標頭全部缺席**
- 位置：`Program.cs:215-226` 管線無任何 `Headers.Add/Append` 或標頭中介軟體。缺 `X-Frame-Options`、`Content-Security-Policy`、`X-Content-Type-Options`、`Referrer-Policy`、`Permissions-Policy`。
- 說明／情境：無 X-Frame-Options/CSP frame-ancestors → 點擊劫持（把控制頁嵌入透明 iframe 誘導操作）；無 CSP → 缺 XSS 縱深防禦；無 nosniff → MIME 嗅探。
- 建議：新增安全標頭中介軟體：`X-Content-Type-Options: nosniff`、`X-Frame-Options: DENY`（或 CSP `frame-ancestors 'none'`）、`Referrer-Policy: no-referrer`；CSP 因現有頁面多 inline script 建議先 `Report-Only` 觀察再收斂。（P1-6）

**[SEC-17] 狀態變更端點全面缺少 CSRF（Antiforgery）防護**
- 位置：僅 `LoginController.cs:47/143` 有 `[ValidateAntiForgeryToken]`；其餘所有狀態變更 API（`/api/control/write`、帳號 CRUD、各 Save/Delete）皆無，全域也未設 `AutoValidateAntiforgeryToken`；`Program.cs` 未顯式設 `Cookie.SameSite`（吃框架預設 `Lax`）。
- 說明／情境：目前實際可利用性偏低（`[FromBody]` JSON + Cookie 預設 `SameSite=Lax` + 無 CORS 政策，三者疊加擋下傳統跨站 CSRF），但屬「靠隱性框架預設值意外被緩解」，非常脆弱：一旦有人為嵌入需求把 `SameSite` 改 `None`，或部署到不強制 SameSite 的嵌入式看板瀏覽器，全部門戶洞開；且同源 XSS(SEC-10) 可直接繞過。
- 建議：全域啟用 `AutoValidateAntiforgeryToken`，前端 AJAX 統一附 `RequestVerificationToken` 標頭；`Program.cs` 顯式設 `Cookie.SameSite`。（P1-6）

**[SEC-18] Web→Engine 控制指令無完整性 / 防重放 ＋ 控制 topic retain 殘留可致重啟誤動作**
- 位置：`MqttRealtimeSubscriberService.cs:621-656`（發佈端 payload 僅 `mid(GUID)/timestamp/value/unit`，明文）對比 `MqttControlSubscribeService.cs:288` 起（接收端不檢查新鮮度、不記已用 mid）；`MqttPublishService.cs`（`PublishRawJsonAsync` 預設 `isRetain=true`）。
- 說明／情境：合法控制路徑雖用 `WithRetainFlag(false)`，但 broker 無 ACL 下攻擊者可自行對 `SCADA/Control/{CID}` 留 `retain=true` 訊息，Engine 每次重連/重啟訂閱時立即收到並執行 → 「開機即誤動作」。側錄合法「啟動泵浦」封包亦可原樣重放。
- 建議：broker ACL 禁止 control 前綴 retain；Engine 收到 retained 控制訊息一律忽略（檢查 `ApplicationMessage.Retain`）；導入簽章 + nonce/序號 + 時間窗防重放。（P2-5）

**[SEC-19] MQTT broker 無認證、無 TLS（明文 1883），其他控制平面 topic 亦可被注入**
- 位置：`MqttConfigModel.cs`（無帳密/TLS 欄位）；四個 MQTT client 皆裸連；`AlarmRuleReloadSubscriber.cs`、`DbCoordinatorReloadSubscriber.cs`、`EnergyLeafBackfillSubscriber.cs`、`LicenseVerifySubscriber.cs` 共用同一匿名 broker。
- 說明／情境：即時量測與控制訊號可被同網段側錄/竄改；攻擊者可注入偽造的規則重載/DB 協調器重載/回填/授權驗證訊息造成服務抖動或 DoS。目前 loopback 風險有限，broker 一旦對外或跨主機即顯著升高。
- 建議：broker 帳密 + TLS(8883) + per-topic ACL；`MqttConfigModel` 增設 `szUsername/szPassword/isUseTls/szCaCertPath`，所有 client 補 `.WithCredentials()`/`.WithTlsOptions()`；帳密由環境變數/密鑰庫注入。（P1-3 / P2-6）

**[SEC-20] Modbus 網段暴露面**
- 位置：`Modbus.json`（設備 `192.168.147.1:502`）；`MqttControlSubscribeService.cs`（每次控制即 `new ModbusTcpClient()` 直連 502）。
- 說明／情境：Modbus TCP 本質無認證無加密；任何能路由到該網段者可直接對 502 下 Modbus 寫入，完全不經本系統。
- 建議：Modbus 段以 VLAN/防火牆隔離，僅允許 Engine 主機來源 IP 存取 502；落實 IT/OT 分區（Purdue model）；設備支援則啟用 Modbus/TCP Security 或前置安全閘道。（P2-6）

**[SEC-21] EmailSetting.json 已進版控，且 Web UI 以明文寫入 SMTP 密碼**
- 位置：`ScadaEngine.Engine/Setting/EmailSetting.json`（git 追蹤中，目前憑證為空）；`ScadaEngine.Web/Services/EmailSenderConfigService.cs:44-51`（寫入邏輯）。
- 說明／情境：Web 後台一旦設定 SMTP，明文 app password 即落入受版控檔案，極易隨下次 commit 外洩。（正面：`Read()` 不回傳密碼明文給前端，作法正確。）
- 建議：`EmailSetting.json` 移出版控（加入 `.gitignore`）、憑證加密或改用密鑰保管庫；CI 加密鑰掃描防誤 commit。（P2-8）

### 🟢 低 (Low)

**[SEC-22] LicenseBridge 硬編碼 HASP vendor code 與明文授權比對字串**

**[SEC-26] 日誌與錯誤處理缺失（稽核軌跡 / 錯誤頁）**

- 說明／情境：資訊洩漏面偏低（生產不顯示堆疊），但錯誤頁 UX 損壞、且缺乏安全稽核軌跡（登入/控制/帳號變更無留存），事故後難以追溯。

**[SEC-27] MqttSetting.json 殘留未使用的連線字串範本（含 sa / TrustServerCertificate）**
- 位置：`ScadaEngine.Engine/MqttSetting/MqttSetting.json`、`ScadaEngine.Web/MqttSetting/MqttSetting.json`（`ConnectionStrings.szDefaultConnection` 範本，runtime 未消費）。

## 七、改善計畫（分階段）

> 工作量標記：S＝小（數小時～1 天）、M＝中（數天）、L＝大（1～數週）。
> 原則：**先止血（降低暴露面、關後門）、再修信任邊界（授權+加密）、後補縱深與治理**。所有變更先在測試環境驗證，生產套用需維護時窗與回滾方案。

### P0 — 緊急止血（目標：72 小時內；多為設定/低碼量，先切斷最易利用路徑）

| # | 項目 | 對應弱點 | 量 |
|---|---|---|:--:|
| P0-1 | **收斂網路暴露面**：Web 監聽由 `0.0.0.0` 改綁內部管理網段/`127.0.0.1`（前置代理時）；防火牆 5038 僅放行必要來源 IP；確認 MQTT broker 綁 loopback、未對外；確認 Modbus 502 不對辦公網開放。（暫時切斷「任何 LAN 節點 → 控制/數據」路徑） | SEC-01/04/05/19/20 | S |
| P0-2 | **關閉預設帳密後門**：確認 DB 已有啟用的強密碼 Admin；以組態開關停用 `admin/admin` fallback（或直接移除常數）。 | SEC-08 | S |

### P1 — 短期（目標：2～4 週；修好核心信任邊界：加密 + 授權 + 控制鏈）

| # | 項目 | 對應弱點 | 量 |

## 九、正面發現（已做對的部分）

平衡呈現——以下是本系統資安體質良好的部分，改善時應保留：

- **SQL 100% 參數化**：全庫 22 個含 Dapper 存取的檔案逐一查核，無任何字串串接/內插注入；無原生 `SqlCommand`/`AddWithValue` 拼字串；Controller 層零 SQL。唯一動態 SQL 為建表 DDL，來源為本機信任設定檔（低風險）。**未發現典型 SQL Injection。**
- **View 端 JSON 注入安全**：全專案無 Newtonsoft，`@Html.Raw(...Json)` 皆走 System.Text.Json 預設編碼器，`<`/`>`/`&` 跳脫為 `<` 等，可擋 `</script>` 突破。
- **Open Redirect 有防護**：登入固定導向 `/ScadaPage`；語系切換用 `Url.IsLocalUrl(returnUrl)` 白名單把關（正面範例）。
- **機密未寫入日誌**：Line token、SMTP 密碼未被記入 log；登入只記帳號不記密碼；`EmailSenderConfigService.Read()` 不回傳密碼明文給前端。
- **多數前端輸出正確跳脫**：`scadapage.js` 各 widget 建構皆呼叫 `escViewHtml`（唯樹節點漏掉，見 SEC-10）。
- **登入表單有 antiforgery**、Cookie `HttpOnly=true`。
- **`LineSetting.json`（真實 token 檔）正確被 `.gitignore` 排除**、歷史 0 次提交。
- **架構分層清晰**：Feature Folder + Controller/Service/Model 分離、DTO 不混在 Controller，利於逐點補授權而不需大改。
